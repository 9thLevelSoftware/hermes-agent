## Part 1 — Subsystem review

### Core agentic loop (turn lifecycle, streaming, tool dispatch, retries, budgets, interrupt/steer, MoA)

The heart of the harness is `run_conversation()` in agent/conversation_loop.py — a fully synchronous ~4,800-line while-loop, extracted from the 12k-LOC `AIAgent` god-class in run_agent.py, that drives one user turn: prologue (agent/turn_context.py builds TurnContext, resets ~15 retry counters, refreshes MCP tools, restores primary provider), then per-iteration API-message assembly with prompt-cache-stable normalization, a nested retry loop wrapping the model call, a large hand-rolled recovery state machine (TurnRetryState one-shot flags + FailoverReason-classified errors), tool dispatch (sequential or thread-pooled concurrent via agent/tool_executor.py), and a post-loop finalizer (agent/turn_finalizer.py) that handles budget-exhaustion summarization, persistence, plugin hooks and background memory/skill review. Four wire transports (chat_completions, anthropic_messages, bedrock_converse, codex_responses) sit behind `agent._get_transport()`, plus an opt-in codex_app_server subprocess runtime that bypasses the whole loop. Everything is thread-based: interrupts are 200ms polling flags fanned out to tool-worker thread IDs and child subagents, streaming runs on daemon worker threads with stale-stream detection, and there is no asyncio anywhere in the core path. The design's strengths are extreme provider-failure hardening (a dozen one-shot recovery ladders, multi-provider fallback chains, credential-pool rotation, adaptive backoff) and prompt-cache discipline; its weaknesses are the monolithic control flow, absence of a permission/approval policy engine, no plan mode, no mid-turn resumability, and flat iteration-count budgeting with no token/cost/wall-clock budgets.

**Gaps vs state-of-the-art harnesses:**
- No permission/approval policy engine in the core loop. Claude Code has permission modes (default/acceptEdits/plan/bypassPermissions), per-tool allow/deny rules persisted in settings, and interactive approval prompts wired into dispatch; Hermes has only thread-local approval callbacks for terminal/sudo, hardcoded guardrail blocks, and plugin pre-tool blocks — there is no policy language, no 'always allow this command pattern' persistence, no per-session permission escalation flow at the tool_executor layer
- No plan mode / hierarchical planning. There is a todo store and skill nudges, but no read-only planning phase with a plan-approval gate before mutating tools unlock (Claude Code plan mode, Codex CLI 'suggest' vs 'auto-edit' vs 'full-auto' approval levels)
- Fully synchronous, thread-and-poll architecture: interrupts are 200ms sleep polls, streaming and Bedrock calls spawn daemon threads joined in 0.3s loops, wedged tool threads are deliberately leaked (abandon_executor). No asyncio, no structured cancellation, no async tool protocol — limits scaling for many concurrent sessions in one process and makes cancellation best-effort
- No mid-turn durability/resume: the loop's control state (retry ladders, continuation counters, pending verification response) lives in Python locals; a process crash mid-turn loses everything except flushed messages. No LangGraph-style checkpointed state graph, no durable execution / replay of an in-flight turn, no turn forking/branching or conversation rewind (opencode /undo, Claude Code /rewind restore both files AND conversation; Hermes CheckpointManager restores files only)
- Budgets are iteration-count only: no per-turn token budget, no cost ceiling (USD) enforcement, no wall-clock turn deadline in the loop itself (cron applies an external 3-min hard interrupt). No adaptive budgeting (e.g. lowering effort as the budget depletes) and no user-visible budget progress
- No tool-argument schema validation before dispatch: only JSON-parse + tool-name checks; hallucinated/missing parameters flow straight into tool code, whereas Claude Code/Gemini CLI validate against the declared JSON schema and return structured validation errors the model can correct cheaply
- Parallelism policy is hardcoded: _PARALLEL_SAFE_TOOLS/_NEVER_PARALLEL_TOOLS frozensets plus a path heuristic for exactly three file tools. No per-tool declared side-effect annotations (MCP-style readOnlyHint/destructiveHint) driving concurrency, batching, or approval requirements
- Error classifier is a giant hand-maintained regex/status/phrase table (1,600 lines, English-phrase matching explicitly acknowledged as brittle in comments); no telemetry-driven or provider-declared retry policy, no per-provider circuit breaker abstraction (only the Nous-specific file-based guard and ad-hoc 60s cooldowns)
- No structured-output mode for final answers: no response_format/JSON-schema enforcement with validate-and-retry in the loop; consumers needing structured results must parse prose
- MoA is advisory-only and latency-additive: reference fan-out gates every tool-loop iteration on cache miss; no debate/voting/consensus, no early-exit when advisors agree, no per-turn MoA spend budget, references cannot call tools, and no async prefetch of references while the aggregator streams
- Steering semantics are coarse: /steer text can only piggyback on tool-result messages (a steer sent before the first tool batch waits indefinitely; one sent after the final answer is returned as 'pending_steer' for the caller to resend). No first-class mid-turn user-message queueing that the model sees as a distinct user turn (Claude Code steering injects real user turns between iterations)
- No per-step model routing: one model runs the whole turn; there is no cheap-model-for-tool-loops / strong-model-for-synthesis routing, no automatic model escalation ladder by failure class beyond the linear fallback chain, and no hedged/raced parallel requests
- Loop detection is limited to identical-tool-call repetition and failure streaks (ToolCallGuardrailController); no semantic/content loop detection on assistant text (Gemini CLI detects repeated content n-grams and self-terminates), no oscillation detection across edits (A→B→A file thrash)
- Observability is bespoke: middleware traces and plugin hooks (langfuse plugin) but no first-class OpenTelemetry spans for iterations/tool calls/retries; no built-in turn replay viewer or trace UI beyond MoA trace files
- The verification stop-loop only checks file mutations (test/lint evidence for changed paths); there is no general self-verification harness (spawn a verifier subagent, run project-configured checks, diff-based review) comparable to Claude Code hooks + verify skills
- Interrupted/timed-out tools return 'effect UNKNOWN' text but there is no transactional undo or automatic state inspection; recovery burden is pushed entirely onto the model
- api_max_retries defaults low and retry accounting is per-API-call: a turn can still stall for many minutes across chain-entries x retries x backoff with no aggregate turn-level retry budget or ETA surfaced to the user
- run_conversation remains a ~4,800-line function with deeply nested one-shot recovery branches; the decomposition campaign (turn_context/turn_finalizer/turn_retry_state) is only Phase 1 — adding new recovery behavior requires editing the monolith and understanding ~30 interacting counters/flags

### Context management: window accounting, compression/compaction, prompt construction, prompt caching, trajectory handling, context references

Hermes routes all context management through a pluggable ContextEngine ABC (agent/context_engine.py) whose default implementation is a 3,200-line lossy-summarization ContextCompressor: it protects a decaying head and a token-budgeted tail, prunes/dedupes old tool results without an LLM, then summarizes the middle window into a 13-section structured handoff via an auxiliary model, with iterative summary updates, focus-topic guidance, extensive failure taxonomy (auth/network aborts, main-model fallback, DB-persisted cooldowns), and anti-thrashing verdicts keyed on provider-reported token counts. Compaction fires from four independent sites (turn-prologue preflight, mid-loop pre-API pressure check, post-response real-usage check, and an 85% gateway session-hygiene safety net) and completes as either a session rotation (new child session id) or in-place SQLite compaction (archive_and_compact soft-archives old rows, keeping them FTS5-searchable). All live accounting is char/4 heuristics (estimate_request_tokens_rough incl. system prompt + tool schemas + flat per-image costs); exact tokenization exists only in the offline trajectory_compressor.py training-data pipeline. Prompt construction is a three-tier (stable/context/volatile) system prompt built once per session, persisted to the session DB, and byte-stable by design to keep Anthropic cache_control prompt caching (system_and_3 layout, 5m/1h TTL, envelope-vs-native layout policy per provider) and local KV-cache prefixes warm; @file/@folder/@git/@url context references expand user messages with deny-list-guarded, budget-capped inline content.

**What exists (capabilities):**
- Pluggable ContextEngine ABC (agent/context_engine.py) selected via config.yaml context.engine; plugins/context_engine/<name>/ loader (plugins/context_engine/__init__.py) supports register(ctx)-style plugins that can register the engine, slash commands, and agent-callable tools (get_tool_schemas/handle_tool_call, e.g. lcm_grep) injected into the tool schema in agent/agent_init.py (~line 1910)

**Gaps vs state-of-the-art harnesses:**
- No exact tokenization anywhere in the live path — every decision (thresholds, tail budgets, summary budgets, preflight, /usage breakdown) runs on a char/4 heuristic with hand-tuned fudge factors; the codebase's own comments admit 30-50% error on code/JSON. No tiktoken, no Anthropic count_tokens endpoint, no Gemini countTokens, no local tokenizer even when one is available (trajectory_compressor.py proves the team can use HF tokenizers but only does so offline). A large fraction of the compressor's complexity (-1 sentinels, awaiting_real_usage latches, defer-to-real-usage heuristics, noisy-estimate carve-outs #36718) is scar tissue compensating for inaccurate counting.
- Compaction is destructive-in-window with no retrieval loop back into it: dropped middle turns survive only as one rolling flat-text summary. There is no mechanism for the model to expand or re-fetch summarized content on demand (no 'expand checkpoint N' tool from the built-in engine; the LCM lossless engine is referenced in docs/tests but NOT shipped — plugins/context_engine/ contains only the loader). session_search FTS5 over archived rows exists as a general tool but nothing auto-links a compaction summary to its archived source rows or teaches the model to query them.
- No tool-result externalization: large tool outputs are truncated or replaced with one-line summaries in place, rather than spilled to disk with a re-readable reference (Claude Code writes big outputs to files and returns paths; opencode similarly). The hook_output_spill mechanism exists but is applied ONLY to plugin pre_llm_call hook outputs, not to ordinary tool results — the obvious generalization is unbuilt.
- No Anthropic server-side context-editing / context-management API integration (clear_tool_uses, context editing beta) — zero hits for it in the codebase — so on Anthropic the harness pays client-side summarization latency and an aux-model call for something the provider can do in-request.
- Fixed, hardcoded caching layout: system_and_3 is the only strategy — no breakpoint on the tool-schema/system boundary vs conversation prefix, no cache-aware placement informed by cache_read/cache_write usage the engine already receives in update_from_response (fields are accepted but the default engine ignores them), no Gemini explicit cachedContent API, no cost/hit-rate telemetry loop to tune TTL or layout.
- Tool schemas are counted (20-30K tokens acknowledged as an 'incompressible floor' that causes compaction thrash, #14695) but never reduced: no deferred tool loading / tool-search-style lazy schemas, no per-turn tool subsetting, no schema compression — toolsets are fixed at session start. State-of-the-art harnesses (Claude Code's ToolSearch deferral, opencode) treat schema weight as compressible.
- Single-shot summarization with per-message truncation: summarizer input clips each message to 6K chars (head 4K/tail 1.5K) and makes ONE LLM call — no map-reduce/chunked multi-pass for very large middles, no fidelity verification of the summary (no self-check pass), and quality silently degrades with each iterative update (the code itself warns users after 2 compactions). No configurable/per-profile compaction prompt template.
- No user-visible checkpoint/rewind of the context state around compaction: in-place mode soft-archives rows (active=0) and rotation preserves the parent session, but there is no /rewind-style UX to restore the pre-compaction window or fork from it (Claude Code Rewind is cited as inspiration for '/compress here' but restore is not implemented; the gateway '--aggressive' LLM-free truncation path is explicitly unsupported, #44794).
- Memory is flat MEMORY.md/USER.md injected wholesale into the volatile tier every turn — no Letta-style tiered core/archival memory with self-editing paged blocks, no size-pressure eviction between memory and conversation, no relevance-scoped memory retrieval per turn (external memory providers can plug in, but the built-in path is a static blob that also breaks the volatile-tier cache whenever it changes).
- No sub-agent context offloading as a compaction strategy: delegate_task exists, but nothing routes context-heavy work (large searches, exploration) into ephemeral child contexts to keep the parent window small, and compaction never considers 'this content belongs to a finished subtask' structure — it is purely positional (head/middle/tail).
- Image accounting is a flat 1500/1600-token constant regardless of dimensions or provider formula (Anthropic w*h/750, OpenAI tiles) — acknowledged in comments as a 'realistic ceiling', so multi-image sessions systematically mis-budget.
- No proactive context forecasting or per-turn budget display: the UI shows percent-used from the last response, but there is no 'this next request will overflow because of X' attribution, no per-tool-result size warnings to the model mid-turn, and no adaptive output-reservation (max_tokens is treated as a static reservation).
- Cross-session lineage is structural only: child sessions chain via parent_session_id and inherit one summary, but there is no multi-ancestor recap synthesis, no lineage-wide search-and-summarize on resume, and resumed sessions rehydrate iterative-summary state only if the handoff text happens to sit in the protected head.
- The @-reference system is one-shot inline injection with no lifecycle: no pinning (re-inject on every turn), no automatic refresh when the referenced file changes, no reference-to-tool handoff for oversized files (it just refuses at 50% of context rather than offering ranged reads), and no IDE-style symbol references (@function/@class) that Gemini CLI/Cursor-style harnesses support.

### Self-improving learning loop: agent-curated memory, curator, learning graph, insights, skill creation/self-improvement, cross-session recall (FTS5), Honcho user modeling

Hermes' learning loop has five cooperating layers. (1) Built-in curated memory: two char-capped, §-delimited markdown stores (MEMORY.md 2200 chars, USER.md 1375 chars) written via a single `memory` tool with atomic batch ops, injected as a frozen system-prompt snapshot for prefix-cache stability, guarded by threat-pattern scanning and a round-trip drift detector. (2) A MemoryManager/MemoryProvider plugin layer (one external provider max) with a rich lifecycle-hook contract (prefetch/sync_turn/on_session_end/on_session_switch/on_pre_compress/on_memory_write/on_delegation) that hosts Honcho (peer-based user modeling with dialectic Q&A), Hindsight, mem0, and a local FTS5+HRR "holographic" fact store. (3) A self-improvement loop: turn/iteration-count nudges fork a whitelisted (memory+skills tools only) copy of the agent after verified turns to patch/create skills and save memories, plus a weekly idle-triggered Curator that deterministically ages skills (stale 30d → archived 90d) and optionally runs an LLM umbrella-consolidation pass with tar.gz snapshots, structured classification (absorbed_into declarations > YAML summary > tool-call audit), and cron-job reference rewriting. (4) Cross-session recall: zero-LLM `session_search` tool over dual FTS5 indexes (unicode + trigram) in the SQLite state DB, with lineage dedup, anchored windows + bookends, cron-source demotion, and cross-profile reads. (5) Presentation layers: a lexical-overlap learning graph for the desktop/TUI "journey" view with user edit/delete mutations, and a descriptive InsightsEngine (tokens/cost/tools/skills/activity). The loop is entirely keyword/heuristic driven — no embeddings, no outcome feedback, and no automatic recall injection from the session DB.

**Gaps vs state-of-the-art harnesses:**
- No semantic/embedding retrieval anywhere in the core loop: skill routing is a static name+description system-prompt index, session recall is FTS5 keyword/BM25 only, and memory→skill graph edges are lexical token overlap. Vector search exists only inside optional external providers (Honcho/mem0/Hindsight). No local embedding index over sessions, skills, or memories; no hybrid BM25+vector ranking; no reranker. Letta, mem0-native harnesses, and even Gemini CLI extensions do semantic recall as table stakes.
- No automatic recall injection from the agent's own history: session_search is purely model-initiated — nothing prefetches relevant past-session context into the prompt (prefetch/queue_prefetch is only implemented by external providers). The builtin provider path has no RAG-over-own-transcripts, so cross-session continuity depends entirely on the model remembering to call a tool.
- Builtin memory is tiny and flat: 2200+1375 chars total, two global files per profile. No hierarchical or paged memory (vs Letta's core/archival split with paging), no per-project/per-cwd scoping (vs Claude Code's directory-scoped CLAUDE.md hierarchy), no namespaces/topics, no per-entry metadata (timestamps, provenance, confidence, TTL/decay), no episodic-vs-semantic separation. Overflow forces destructive consolidation instead of eviction-to-archive (archive_on_overflow exists but defaults off and archived content is not searchable).
- No outcome/utility feedback loop: skill telemetry counts views/uses/patches but never whether a loaded skill actually helped; no success/failure signal wired from turn outcomes back into skill or memory scoring (holographic trust scores are optional and provider-local); the curator judges by content overlap and staleness, never utility. No eval/regression harness for skills — background-review edits ship unvalidated (the review fork's tool whitelist excludes terminal/execute, so a learned procedure can never be verified by running it).
- Learning triggers are blind counters: background review fires every N turns/iterations, not on detected user corrections, task failures, repeated retries, or explicit feedback. No frustration/correction classifier, no thumbs-up/down capture, no 'this session went badly — review it' path. State-of-the-art harnesses increasingly trigger reflection on failure signals.
- The learning graph is display-only: edges are never used for retrieval, skill suggestion, or curation input; no co-usage mining from session history, no temporal/causal edges, top-4 lexical cap. A genuine knowledge graph (typed relations, traversal-based recall) exists only in the optional Hindsight provider.
- One-external-provider limit with no composition: you cannot run Honcho user modeling alongside the local holographic fact store; there is no memory router/composite provider and no policy layer for which store answers which query type.
- No session summarization index: FTS5 returns raw message rows; there is no per-session distilled summary artifact, topic/decision extraction, or auto-generated searchable digest (the earlier summary-LLM path was deliberately removed in the session_search merge). 'What did we decide about X' works only if the exact keywords appear in a message.
- Curator consolidation is opt-out-by-default, monolithic, and unattended: one giant fork with max_iterations=9999, no human-in-the-loop approval queue (dry-run exists but is manual and all-or-nothing), no incremental per-cluster passes, no diff preview before archive/patch. Background-review skill writes similarly land silently with only after-the-fact summaries — no staged-approval flow like the memory tool's write_approval path.
- No sharing/export of learned knowledge: agent-created skills cannot be published to the hub, synced across profiles/machines (backup/restore only), or written into per-repo skill directories (skills.external_dirs is read-only to the learning loop). No team/multi-agent knowledge exchange, vs Claude Code plugin skills and opencode project-local skill conventions.

### Tool system: builtin tools, toolset composition/distribution, tool metadata, guardrails, MCP client+server, RPC scripting (execute_code)

### Multi-platform gateway (transports, relay, hooks, voice, continuity, notifications)

The gateway is a single long-lived asyncio process (gateway/run.py, ~21k LOC GatewayRunner composed of authz/kanban/slash mixins) that normalizes ~30 messaging platforms into one MessageEvent/SendResult contract defined by BasePlatformAdapter (gateway/platforms/base.py, ~5.6k LOC). Heavy platforms (Telegram, Discord, Slack, WhatsApp bridge, Matrix, etc.) live as lazily-loaded plugins under plugins/platforms/* self-registering into gateway/platform_registry.py; Signal, WhatsApp Cloud, BlueBubbles/iMessage, a generic HMAC webhook platform, and an OpenAI-compatible API server are in-core. **Gaps vs state-of-the-art harnesses:**
- Hook system is far weaker than Claude Code/Codex/Gemini hooks: gateway hooks are fire-and-forget lifecycle events with 500-char-truncated context; there is no blocking PreToolUse/PostToolUse-style hook at the gateway boundary that can allow/deny/mutate a tool call or an outbound message, no shell-command hook runners with a JSON decision contract, no matchers/timeouts/config-file registration (only ~/.hermes/hooks dirs), and gateway/builtin_hooks/ is an empty stub — the declared extension point has zero shipped hooks
- No inbound reaction events: every adapter can SET reactions, but a user reacting to a bot message (👍 to approve, ❌ to cancel, emoji feedback for the learning loop) is never surfaced as a MessageEvent on any platform — no MessageReactionHandler (Telegram) or on_raw_reaction_add (Discord) is registered. This is a missed lightweight approval/feedback/RLHF-signal channel, notable for a self-improving harness
- User message edits and deletions are ignored: no edited_message (Telegram) or on_message_edit/on_message_delete (Discord) handling, so there is no 'edited prompt → rerun/correct' semantics that modern chat surfaces (and the TUI itself) support; delete-based retraction of context is also impossible
- Voice is batch-only: STT transcribes complete clips via provider HTTP APIs; there is no streaming ASR with partial transcripts, no VAD-driven barge-in/interruption in Discord voice, no wake word, and no realtime speech-to-speech (OpenAI Realtime / Gemini Live style) integration; TTS in DMs is per-message file synthesis rather than streamed playback
- Cross-platform continuity is one-directional and coarse: /handoff exists only CLI→gateway-home-channel; there is no platform→platform migration command (Telegram thread → Discord thread), no unified cross-platform user identity (identity is strictly per-platform; pairing approves per-platform IDs with no account-linking table), and mirror.py mirrors text only (media and tool events are not mirrored), so 'continue this conversation on Signal' works only via home-channel + transcript rebind
- No persistent outbound delivery queue: sends that fail mid-restart or while a platform is paused are lost (dead_targets registry only suppresses known-dead chats); state-of-the-art notification systems have durable outboxes with retry/backoff per target — cron delivery relies on the process being alive and _send_with_retry's in-memory attempts
- Interactive UI components are ad hoc per adapter: inline keyboards/model pickers/clarify buttons exist for Telegram and partially Slack (block_kit.py is Slack-only) and Discord slash options, but there is no cross-platform component schema (buttons/forms/cards) in the adapter contract — clarify/approval degrade to typed text on most platforms, and the relay CapabilityDescriptor has no fields for interactive components, reactions, or media capabilities
- Notification policy model is thin: no quiet hours, priority levels, digest/batching, per-user or per-channel notification preferences, or dedup across channels — background_process_notifications is a single global mode (all/result/error/off); compare Letta/LangGraph schedulable notification policies or Claude Code's granular notification hooks
- Relay is experimental and non-reproducible from this repo: the connector side ('gateway-gateway', relayProvision.ts etc.) lives in a separate private repo, the descriptor schema is explicitly unstable (CONTRACT_VERSION 1, breaking-change caveats), only Class-1 basics (send/edit/typing/interrupt) are wired — no media, no threads-over-relay, no interactive components — so self-hosters cannot actually use the multi-tenant path
- Group-chat multi-agent coordination is absent: exclusive_bot_mentions and ALLOW_BOTS are filters, not coordination; there is no protocol for multiple Hermes instances (or Hermes + other bots) sharing a channel with turn-taking, no A2A/agent-to-agent addressing over the messaging surface beyond raw text, while the kanban dispatcher handles this only for internal work items
- No replayable platform simulator: adapters are tested with unit mocks; there is no 'fake platform' transport for end-to-end gateway testing or record/replay of platform traffic (opencode and Claude Code maintain replay harnesses), making the 8k-line Telegram/Discord adapters risky to modify
- Observability of the message pipeline is log-based only: memory monitor and shutdown forensics exist, but there is no OpenTelemetry tracing across inbound→agent→delivery, no per-platform delivery/latency/error-rate metrics endpoint, and no structured event log for message lifecycle (queued/debounced/delivered/failed) that a dashboard could consume
- Hooks/webhooks cannot inject scheduled or conditional behavior per conversation: no per-chat automation rules (e.g. 'summarize this channel nightly' as a first-class binding — must be hand-built with cron + deliver targets), no channel-level memory/context policies beyond channel_prompts and skill bindings
- i18n covers only ~414 static strings in 16 catalogs; slash-command descriptions, /help output, most gateway notices, and all adapter error messages remain English-only, and there is no per-chat language preference (resolution is global config/env)
- Media pipeline gaps: inbound documents/images are cached to local paths for vision, but there is no media size/type policy layer, no virus/malware scanning hook, no cross-platform media transcoding service (each adapter hand-rolls format constraints like Telegram sendAudio MP3/M4A), and stickers get vision descriptions only on Telegram

### Model/provider layer: provider adapters, model metadata/registry, credential pooling, rate limiting, model switching, reasoning support

Hermes routes every LLM call through a two-layer design: declarative ProviderProfile plugins (providers/base.py, discovered from plugins/model-providers/<name>/ with $HERMES_HOME user overrides) describe per-provider quirks, while four hardcoded ProviderTransport implementations (agent/transports/: chat_completions, anthropic_messages, codex_responses, bedrock_converse) own message/tool conversion and response normalization into a shared NormalizedResponse/ToolCall type. Client lifecycle, streaming, retries, credential rotation, and failover stay on the synchronous AIAgent (run_agent.py + agent/chat_completion_helpers.py + agent/conversation_loop.py), driven by a ~20-value FailoverReason classifier with recovery hints. Model metadata comes from a models.dev snapshot/cache (4000+ models: context, cost, capabilities) layered with OpenRouter live metadata, provider /models probing waterfalls, and an elaborate error-driven context-length resolution chain; credentials live in a persistent multi-entry per-provider pool with OAuth refresh, exhaustion TTLs, and four rotation strategies. The layer is unusually broad (30+ providers, 5 OAuth flows, subprocess backends for Codex app-server and Copilot ACP) but has three overlapping provider registries, is sync-only, and lacks budget enforcement, proactive rate limiting, structured output, and batch APIs.

### Execution environments & security (terminal backends, file safety, secret scoping, approvals, SSL guard, redaction)

Hermes runs LLM shell commands through a unified spawn-per-call BaseEnvironment abstraction (tools/environments/) with six backends — local, Docker (cap-drop ALL + no-new-privileges + tmpfs + pids-limit), SSH (ControlMaster + tar file-sync), Singularity (--containall + overlay persistence), Modal (SDK sandboxes + filesystem snapshots), Daytona (SDK sandboxes) — each preserving env/cwd across calls via an atomic bash snapshot file. Security is explicitly layered heuristics, not a boundary (SECURITY.md §2.2): a 3,900-line approval module (tools/approval.py) combines an unconditional hardline blocklist, ~70 dangerous-command regexes with shell-deobfuscation variants, an aux-LLM 'smart approval' guardian, the external Tirith scanner, gateway blocking-approval queues, and cron/yolo/deny policy; secrets are scoped via a derived provider-env blocklist, a fail-closed multiplex ContextVar scope, a pluggable SecretSource registry (Bitwarden/1Password), and regex redaction of all tool output. The 9thLevelSoftware fork's cap-review PRs hardened approvals into a durable, integrity-checked system: unresolved requests persist to ~/.hermes/approval_requests.json (0600, safe-fields-only) keyed by request_id with SHA-256 argument hashes and identity binding (operation/tool/policy_key/requester/channel) that fail closed on any mismatch, and the state file itself was added to both the terminal dangerous-pattern set (sed/perl/tee/cp/mv/ln, profile paths, dot-segments) and the file_tools write deny. The dominant architectural fact — and the source of most gaps — is that everything is blocklist/regex-based screening inside the agent process, with no OS-level sandboxing of the default local backend and no default-deny permission rules engine.

**Gaps vs state-of-the-art harnesses:**
- No OS-level sandboxing for the default local backend: no Linux Landlock/seccomp, no bubblewrap, no macOS Seatbelt/sandbox-exec profile, no Windows job objects. Claude Code (sandbox runtime with FS+network profiles) and Codex CLI (landlock+seccomp / Seatbelt, workspace-write vs read-only policies) sandbox even local execution; Hermes local runs raw bash as the user and SECURITY.md openly concedes every in-process guard is a heuristic. This is the single biggest gap and the root of several below.
- Blocklist-only permission model — no default-deny rules engine. There is no equivalent of Claude Code's allow/ask/deny permission rules with tool-pattern matching (e.g. Bash(git:*)) layered across enterprise/project/user settings, Codex's approval_policy=untrusted/on-request/never, or opencode's per-tool permission config. A novel destructive command that matches no regex runs with zero friction; there is no mode where every command not explicitly allowlisted requires approval, and no project-scoped (.hermes/ per-repo) permission file — the permanent allowlist is one global config.yaml list keyed by pattern description strings.
- Command analysis is regex over ad-hoc normalization variants, not shell AST parsing. No bashlex/tree-sitter decomposition of compound commands (`cd x && curl … | sh` is matched as one string), so both false negatives (unmodeled obfuscations) and false positives (patterns matching data arguments) are structural; Claude Code and Codex both parse commands into trees before matching prefixes/verbs.
- No built-in network egress policy for execution environments: docker_network is a boolean all-or-nothing; the egress-isolation story is a hand-rolled squid + docker-compose recipe in docs/security/network-egress-isolation.md. No domain-allowlist proxy auto-injected into sandboxes (Claude Code sandbox allowedDomains, Codex sandbox network disabled by default), no per-command network toggle, and the SSRF guard (tools/url_safety.py) covers only the web tools — terminal/execute_code curl can reach anything.
- File-tool writes are ungated by default: write_file/patch execute immediately anywhere outside the sensitive-path denylist — no per-edit diff preview + accept/reject flow (the default UX in Claude Code, Gemini CLI, opencode). The approval gate covers dangerous shell patterns, plugin escalations, and memory/skill writes (write_approval.py), but arbitrary project-file mutation is trusted. HERMES_WRITE_SAFE_ROOT (write jail) is an opt-in env var, not a first-class per-project setting.
- Non-interactive contexts FAIL OPEN for dangerous commands: when neither interactive CLI nor gateway session is detected (bare scripts, misconfigured embeddings), _run_approval_gate auto-approves with only a log warning (fail_closed_when_no_human defaults False for the dangerous-command path; only plugin escalations opt into fail-closed). Headless Claude Code/Codex default to deny-without-explicit-grant.
- Terminal-backend isolation doesn't cover non-shell code paths (acknowledged in SECURITY.md): MCP subprocesses, plugin/skill Python code, and hook dispatch run in the host agent process with full privileges even when TERMINAL_ENV=docker. Skills are regex/AST-scanned at install then imported with full trust — no subprocess/capability isolation for plugin execution (vs. harnesses that keep extensions out-of-process or WASM-sandboxed).
- Persisted approval integrity stops at path-deny heuristics: approval_requests.json (and config.yaml's allowlist) have no HMAC/signature or separate privilege domain, so any write primitive not covered by the regex/file-tool denies (python -c via execute_code on local, a skill's own Python, an MCP server) can forge resolved approvals or append allowlist entries. The fork hardened the path-matching but the trust anchor is still 'same OS user can't write the file', which is false in-process.
- Redaction is shape-based only — no value-based masking of actually-loaded secrets: the redactor never registers the concrete values from ~/.hermes/.env / secret sources / auth.json and masks occurrences, so any credential lacking a known prefix or KEY=value shape (opaque tokens echoed bare, split across lines, base64'd) leaks into transcripts/logs; web-URL query params are deliberately passed through, leaving a documented exfil-visible channel.
- Secret lifecycle is startup-only: SecretSource contract is explicitly read-only, synchronous, once-per-process — no mid-session refresh/rotation, no short-lived credential minting or leasing per tool call, no write-back ('save to vault'), no OS keychain backend (macOS Keychain/libsecret/Windows Credential Manager) for Hermes' own plaintext 0600 .env, and no stale-cache fallback policy despite the ErrorKind taxonomy anticipating one. Bitwarden's disk cache stores plaintext secret values (bws_cache.json) mitigated only by a read-deny.
- No structured security audit trail: approval grants/denies, hardline blocks, and executed dangerous commands surface only as scattered logger lines; there is no tamper-evident or even dedicated audit log/JSONL of security decisions, no OTel security events, no session-level 'what was approved and by whom' review surface (Claude Code session transcripts + hooks; enterprise harnesses ship audit sinks).
- SSH backend is thin on security: StrictHostKeyChecking=accept-new (TOFU) with no host-key pinning, no jump-host support, and FileSyncManager pushes the host's ~/.hermes credentials/ and skills/ directories onto every remote — broadening credential exposure to any SSH target rather than scoping secrets per environment; same wholesale ~/.hermes sync goes to Modal/Daytona cloud sandboxes.
- No resource limits on the local backend: no ulimit/cgroup/nice wrapper around local commands (timeout only), so fork bombs that dodge the one classic regex, memory hogs, and disk fills are unbounded; Docker limits themselves silently degrade to none when the cgroup probe fails (warning only). No per-command disk/network quota anywhere.
- Container supply chain unpinned: default sandbox image is a third-party tag (nikolaik/python-nodejs:python3.11-nodejs20) with no digest pinning or signature verification (ironically Tirith's own binary IS cosign-verified); container reuse deliberately ignores image/mount/resource drift on label match.
- No read-only / plan-mode execution posture: nothing like Claude Code plan mode or Codex read-only sandbox — no way to run a session where mutating commands are structurally blocked (the new read_only/destructive registry metadata exists but no policy middleware consumes it yet — it's substrate without an engine).
- Approval scoping granularity is coarse: 'always' approvals are keyed by pattern-description globally (not per project/directory/host), session approvals by session_key only; no TTL on permanent allowlist entries, no per-backend or per-cwd scoping, and no UI to review/revoke granted permissions beyond hand-editing config.yaml.
- Smart-approval guardian is a single one-shot aux-LLM call returning one word — no structured risk rubric, no command-context (cwd, recent history, env type) provided, no dual-model or self-consistency check, and its own prompt-injection resistance rests on comment-stripping + delimiters; the guarded command length is uncapped.
- Windows is a second-class security citizen: Tirith unsupported (regex-only), no sandboxing primitives, and several guards (process-group kill, fcntl allowlist locking) run in degraded best-effort modes.

### Delegation and automation: subagent spawning/isolation/lifecycle, background review, cron scheduler and natural-language automations, batch trajectory generation, SWE runner

Hermes runs delegation as in-process, thread-based child AIAgent instances spawned by the delegate_task tool (tools/delegate_tool.py, ~3.6k lines): children get fresh conversations, intersected/blocklisted toolsets, optional leaf/orchestrator roles with a configurable spawn-depth cap, per-config provider/model credential overrides, and heavy observability (live spawn-tree registry, progress relay events, cost/token rollup into the parent, summary budgeting against parent context headroom with disk spill). Model-initiated delegations are background-by-default: tools/async_delegation.py owns a daemon executor plus a persisted delegations.json registry whose completion events re-enter the conversation through the shared process_registry.completion_queue drained by both CLI and gateway. Automation is a separate, mature cron stack (cron/jobs.py store + cron/scheduler.py 60s ticker with a pluggable CronScheduler provider ABC) that spawns a full AIAgent per job with injection scanning, wake-gate scripts, no_agent script-only mode, per-job model/toolset/workdir overrides, and delivery into live gateway platform adapters — layered with blueprint skills, a curated suggestion catalog, and natural-language schedules. Self-improvement runs as post-turn forked review agents (agent/background_review.py) with a memory/skill tool whitelist and prompt-cache parity tricks, and offline data generation lives in batch_runner.py (multiprocessing, checkpoint/resume, toolset distributions) and mini_swe_runner.py (single-terminal-tool loop over local/Docker/Modal envs emitting Hermes trajectory format). An adjacent kanban dispatcher (hermes_cli/kanban_db.py) spawns detached `hermes chat -q` subprocess workers per card via SQLite CAS claims — the only OS-process-isolated multi-agent path in the repo.

**Gaps vs state-of-the-art harnesses:**
- No user-definable named subagent types/personas. Claude Code has .claude/agents/*.md (per-agent system prompt, tool allowlist, model); opencode has agent configs. Hermes has only leaf/orchestrator, and the model surface was deliberately stripped: per-call toolsets and model selection are removed from delegate_task (toolsets=None forced in delegate_task(); model pinned only globally via delegation.provider/model in config.yaml). There is no registry of reusable, project-scoped subagent definitions the model can pick from.
- Subagents are threads in the parent process, not isolated processes/sandboxes. They share Python globals (the model_tools._last_resolved_tool_names save/restore dance in delegate_task is a symptom), share the GIL, share env vars, and have no per-child cwd/worktree/container isolation (only a separate terminal-session task_id). Contrast: Claude Code subagents get separate contexts + optional worktrees; the repo's own kanban dispatcher already does subprocess isolation but delegate_task doesn't reuse it.
- No mid-flight steering or bidirectional communication with a running subagent: the parent can only interrupt. There is no 'send follow-up instruction to child', no child→parent question channel (clarify is blocklisted with no queued-question alternative), and no shared blackboard/message bus between sibling subagents (kanban swarm's blackboard is not available to delegate_task children).
- Background delegations are not durable: process exit discards in-flight subagent work (records only flip to 'interrupted'; the tool description itself tells the model to use cronjob for durable work). No checkpoint/resume of a child conversation, no re-attach after gateway restart — unlike Codex cloud tasks or CCR-style persistent sessions. The delegations.json record preserves goal/context for manual redispatch only.
- No task graph for delegation: batch tasks are fully independent, join-all semantics only. No inter-task dependencies, no pipelining (child A output → child B input), no map-reduce or planner-executor scaffolds. The workflow graph kernel (tools/workflow_tools.py + hermes_cli/workflows_*) exists but is a separate opt-in kanban-backed subsystem not wired into delegate_task, and its agent nodes just materialize kanban cards.
- No verification/critic layer on subagent output: summaries are explicitly self-reports (the schema warns the model to verify handles itself). No automatic verifier pass, no LLM-judge or test-execution gate on child results, no retry-with-feedback loop on failed/low-quality summaries — areas where SOTA harnesses (Claude Code verify flows, SWE-agent-style evaluators) are ahead.
- Coarse permission model for delegated work: delegation.subagent_auto_approve is a single global bool (auto-deny vs auto-approve dangerous commands). No per-tool/per-path/per-command allowlists per subagent, no inheritance of a structured permission policy like Claude Code settings.json permission rules; cron similarly runs effectively auto-approved (HERMES_CRON_SESSION cron_mode).
- No per-delegation or per-tree cost/token budgets: cost is measured and rolled up but never enforced. Only max_iterations caps a child; no USD/token ceiling for a delegation tree, a background pool, or the cron fleet — a runaway orchestrator at raised depth multiplies cost with no brake beyond concurrency caps.
- Stuck-child handling is passive: the heartbeat staleness monitor just stops refreshing parent activity so a gateway timeout can fire. No automatic retry/respawn of a wedged child, no partial-result salvage from a timed-out child (its in-progress transcript is discarded; only session DB rows remain).
- Weak post-hoc inspectability of finished subagents: child sessions are deliberately excluded from session pickers (_delegate_from marker), spawn-tree events are ephemeral (live TUI only), and there is no `inspect/replay delegation <id>` command that reconstructs a finished child run — versus Claude Code's transcript/resume of subagent output.
- Cron trigger model is time-only: 60s tick granularity, single configured timezone (no per-job TZ), no jitter/spread, no event triggers (file-watch, inbound webhook, email/PR events) in the built-in — event-driven automation exists only as the poll-style wake-gate script pattern or a hypothetical external provider (Chronos contract is explicitly experimental with one consumer).
- Background review/learning loop is fire-and-forget: one daemon thread per qualifying turn, lost on process exit, no queue/dedup/batching, no offline catch-up pass over sessions that never got reviewed, and no A/B or quality scoring of the skill edits it makes (curator consolidates but doesn't evaluate skill efficacy).
- Datagen stack lacks evaluation: batch_runner filters only on 'reasoning present'; no reward model/judge scoring, no environment-verified success signal (mini_swe_runner has no test-harness grading a la SWE-bench), no automatic dataset upload/versioning; mini_swe_runner also duplicates a hand-rolled LLM loop instead of reusing AIAgent, so it silently drifts from the main harness (its own temperature shim hints at this).
- No remote/cross-machine delegation: subagents, async delegations, and background reviews all run on the local machine. The only remote execution is terminal backends (Docker/Modal/SSH) for commands, not for whole agents; nothing like Codex cloud runners, Claude Code on the web, or LangGraph platform deployments for offloading a delegation.
- Delegation concurrency primitives are ad hoc: reject-not-queue at capacity means the model must handle fallback; there is no work queue with priorities, no scheduling fairness across sessions in a shared gateway, and the DelegateEvent lifecycle enum reserves TASK_SPAWNED/COMPLETED/FAILED that are 'not currently emitted' — the formal event model is half-built.

#### Anthropic Context Management API: server-side context editing + server-side compaction (compact_20260112)

A `context_management.edits` parameter on the Messages API with three server-side strategies: (1) `clear_tool_uses_20250919` — clears oldest tool_use/tool_result pairs past a token trigger (defaults: trigger 100k input tokens, keep last 3, with `exclude_tools`, `clear_at_least`, `clear_tool_inputs` knobs); (2) `clear_thinking_20251015` — prunes extended-thinking blocks keeping last N assistant turns; (3) `compact_20260112` (new beta header `compact-2026-01-12`, shipped January 2026) — the API itself summarizes older history into an opaque `compaction` content block when input tokens cross a trigger (default 150k, min 50k), with `pause_after_compaction` (stop_reason "compaction" so the harness can inject state before continuing), custom `instructions` for the summary prompt, per-iteration billing via `usage.iterations`, and `cache_control` supported on compaction blocks. Editing happens server-side before token counting and after cache lookup; clients keep their full unmodified history. Anthropic reports 84% token savings and 39% performance improvement on a 100-turn agentic benchmark with editing+memory. The old client-side `compaction_control` SDK helper is now deprecated in favor of the server-side edit. ZDR-eligible.

**Why it matters for this harness:** This moves the hardest harness problem — deciding what to evict and when — behind the API while preserving prompt-cache prefixes, which client-side stripping destroys. For Hermes Agent it means near-infinite chat-platform conversations without hand-rolled summarization for Anthropic-served models; the harness only needs to pass `context_management` kwargs and round-trip compaction blocks. Because it's Anthropic-only, a multi-provider harness needs a provider-capability layer: use server-side editing/compaction when talking to Anthropic (directly or via Bedrock/Vertex), fall back to client-side compression elsewhere — exactly the design discussed in Hermes Agent issue #526.

**Maturity:** Shipped, public beta (headers `context-management-2025-06-27` and `compact-2026-01-12`). Context editing live since Sep 29, 2025; server-side compaction since Jan 2026. Available on the Claude API, Amazon Bedrock, and Google Vertex AI. Supported on Claude Fable 5, Mythos 5, Opus 4.6-4.8, Sonnet 4.6/5.

#### Prompt-cache-aware harness design (Manus KV-cache doctrine + 'Don't Break the Cache' evaluation)

A now-canonical set of harness rules from Manus's context-engineering post: KV-cache hit rate is the single most important production metric (agent input:output ratio ~100:1); keep the prompt prefix byte-stable (no timestamps), make context append-only with deterministic JSON serialization, and 'mask, don't remove' tools — constrain tool choice via logit masking/state machine instead of editing tool definitions, which invalidates the cache. The economics: Anthropic explicit caching reads cost 0.1x input (writes 1.25x for 5-min TTL, 2x for 1-hour TTL); OpenAI implicit caching gives 50% off cached prefixes over 1,024 tokens with no write premium. A January 2026 arXiv paper, 'Don't Break the Cache: An Evaluation of Prompt Caching for Long-Horizon Agentic Tasks' (arXiv:2601.06007), empirically quantifies how eviction/edit strategies trade cache invalidation against context savings on long-horizon tasks — the formal treatment of the tradeoff Anthropic exposes via `clear_at_least`.

**Why it matters for this harness:** For a self-funded open-source agent on VPS/serverless, cache discipline is the difference between $3/MTok and $0.30/MTok on the dominant token flow. Concretely for Hermes: pin system prompt + tool schemas as a stable cached prefix per provider; never mutate history in place (let server-side editing do it, since it runs after cache lookup); avoid dynamic tool loading mid-conversation; and treat any client-side compression as a deliberate cache-write event, not a free operation.

**Maturity:** Established practice: Manus post July 2025, widely canonized; provider caching shipped and stable (Anthropic explicit breakpoints w/ 5-min and 1-h TTLs; OpenAI automatic). The evaluation paper is Jan 2026 research.

**Sources:** https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus · https://arxiv.org/html/2601.06007v2 · https://rlancemartin.github.io/2025/10/15/manus/

#### Subagent-as-context-firewall: isolated subagent contexts returning compact summaries (now a convergent standard)

Anthropic's 'Effective context engineering for AI agents' (Sep 29, 2025) formalized the attention-budget framing: attention is a depleting resource subject to 'context rot' (recall degrades as tokens grow), so harnesses should spend it deliberately — sub-agents do deep exploration in clean, isolated context windows and return only 1,000-2,000-token structured summaries to the orchestrator; intermediate tool calls never touch the parent context. An April 2026 Arize cross-harness study of Pi, OpenClaw, Claude Code, and Letta found ALL four isolate subagent sessions by default (passing only task descriptions, not parent history), all hard-cap file reads (e.g. Claude Code: 256KB stat-checked cap, 25k-token post-read budget), all cap tool-result sizes, and all support offset/limit pagination — convergent evolution on the same bounded-working-set solution. Claude Code adds an experimental context-fork mode.

**Why it matters for this harness:** Hermes' research-y tasks (web search, codebase exploration, email triage) generate exactly the noisy intermediate tokens this pattern quarantines. Implementing spawn-subagent-with-clean-context + summary-only-return gives Hermes (a) effective context multiplication, (b) protection of the orchestrator's cached prefix, and (c) a natural seam for its self-improving loop — subagent summaries are ideal material for memory distillation. The convergence across four independent harnesses means this is de-risked, copyable design, with published caps/budgets to crib.

**Maturity:** Shipped and standard across major harnesses (Claude Code subagents GA; Pi, OpenClaw, Letta). The attention-budget/context-rot framing is published guidance from Anthropic (Sep 2025) building on Chroma's context-rot research.

**Sources:** https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents · https://arize.com/blog/context-management-in-agent-harnesses/

#### Microcompaction: hot-tail / cold-storage tool-result offloading (Claude Code internals)

A three-layer client-side design inside Claude Code, documented in detail by a Jan 21, 2026 reverse-engineering deep dive: (1) microcompaction — when tool outputs (Read, Bash, Grep, WebFetch, etc.) get large, older results are written to disk and replaced in context by a retrievable path reference, keeping only a 'hot tail' of recent results inline; (2) auto-compaction — headroom accounting (reserve space for output + the summarization itself), growth-rate-based rechecking, and a 9-section structured summary checklist (user intent, decisions, files touched, errors+fixes, pending tasks, next steps); (3) post-compaction restoration — re-read the 5 most recently accessed files, preserve todo list and plan state, inject a continuation message. OpenClaw adds a related 'pre-compaction flush' hook letting the agent persist state to memory files before history is destroyed; Codex CLI/OpenCode have parallel designs.

**Why it matters for this harness:** This is the highest-leverage copyable technique for Hermes as a client-side harness: it works with every provider (unlike server-side APIs), and a VPS gives free disk for cold storage. Offloading bulky tool results early means most sessions never hit full compaction at all, and the structured-summary + restore-recent-files recipe fixes the 'amnesia after compaction' failure mode. The pre-compaction flush hook is the natural integration point with a memory system.

**Maturity:** Shipped in production (Claude Code; comparable mechanisms in Codex CLI, OpenCode, OpenClaw). The internals documentation is third-party reverse engineering (Jan-Apr 2026), not official docs.

**Sources:** https://decodeclaude.com/compaction-deep-dive/ · https://codex.danielvaughan.com/2026/04/14/context-compaction-deep-dive-codex-cli-claude-code-opencode/ · https://arize.com/blog/context-management-in-agent-harnesses/

#### Context quarantine for untrusted content: CaMeL dual-LLM pattern and its 2026 operationalization

Google DeepMind's CaMeL ('Defeating Prompt Injections by Design', arXiv:2503.18813) splits the agent into a Privileged LLM (plans, calls tools, never sees untrusted data) and a Quarantined LLM (parses untrusted tool output, has no tool access), with a custom Python interpreter tracking data provenance via capability metadata and enforcing policy before each tool call; it blocked ~67% of AgentDojo attacks with provable guarantees for the rest of the design. Since then: 'Operationalizing CaMeL' (arXiv:2505.22852) hardened it for enterprise; 2026 research extends it (untrusted content masking for web agents where the main agent queries masked elements at question-time, arXiv:2607.05277; AgentSys hierarchical memory isolation, arXiv:2602.07398); and the pattern is now codified in the OWASP AI Agent Security Cheat Sheet — the quarantined worker gets minimal context (no long-horizon memory, no conversation history, not even the original user query) and returns only a compact structured object.

**Why it matters for this harness:** Hermes is a personal agent reading email, web pages, and chat messages — the textbook indirect-prompt-injection target — and its self-improving memory makes injection persistent (a poisoned memory file attacks every future session). Context quarantine is the same mechanism as subagent context isolation applied as a security boundary: route untrusted tool output through a schema-constrained, tool-less worker before it enters the orchestrator's context. This is cheap to retrofit into a harness that already has subagents.

**Maturity:** Research maturing toward practice: CaMeL is a published DeepMind design (Apr 2025) with 2025-2026 operationalization papers; the dual-LLM/quarantine pattern itself is shipped in security-conscious harnesses and standardized in OWASP guidance. No single turnkey OSS implementation dominates yet.

**Sources:** https://simonwillison.net/2025/Apr/11/camel/ · https://css.csail.mit.edu/6.5660/2026/readings/camel.pdf · https://arxiv.org/html/2505.22852v1

#### Letta memory blocks + sleep-time compute (background memory reorganization)

Letta (the MemGPT successor) structures the context window as labeled, size-capped 'memory blocks' that the agent self-edits with memory tools, and that can be shared across multiple agents. Its 'sleep-time compute' architecture runs a separate sleep-time agent in the background during idle periods that shares the primary agent's memory blocks and asynchronously consolidates archival items, rewrites messy blocks, and distills recent conversations into stable notes — moving memory maintenance off the critical path (vs. MemGPT's lazy in-conversation updates). Letta also embeds attached files into vector stores for semantic search alongside raw access, and compacts at 90% of the context window.

**Why it matters for this harness:** This is the most complete blueprint for Hermes' 'self-improving personal agent' goal: memory blocks map cleanly onto a per-user profile section of the system prompt (stable → cacheable), shared blocks enable one memory across chat platforms and subagents, and sleep-time compute is tailor-made for a VPS agent that sits idle between chat messages — burn cheap idle cycles distilling the day's conversations into durable memory instead of doing it synchronously mid-reply. It also pairs naturally with cron/routine triggers Hermes already has.

**Maturity:** Shipped: open-source Letta framework + Letta Cloud; sleep-time agents documented as a supported architecture; memory blocks are core GA API. Sleep-time compute originated as a Letta/UC Berkeley research paper (2025) and is productized.

#### Sleep-time compute: offline memory consolidation with measured 5x cost / +13-18% accuracy gains

Published technique (arXiv 2504.13171, 'Sleep-time Compute: Beyond Inference Scaling at Test-time', Lin, Snell, Packer, Wooders, Stoica, Gonzalez — Letta + UC Berkeley, April 2025): the model 'thinks' about its context during idle time before queries arrive. Results: ~5x reduction in test-time tokens for equivalent accuracy, up to +13% (Stateful GSM-Symbolic) and +18% (Stateful AIME) accuracy when scaled, and 2.5x per-query cost amortization across related queries. Shipped in Letta as sleep-time agents — background agents that share the primary agent's memory blocks and reorganize them asynchronously. Honcho's 'dreaming' (prune, consolidate, deduce) is the same pattern independently shipped.

**Why it matters for this harness:** A chat-platform personal agent is idle most of the day — that idle VPS/serverless time is free compute for memory consolidation. Instead of lazy incremental memory writes during conversation (latency cost), Hermes can run a scheduled background pass that compresses episodic logs into consolidated memory, pre-computes likely-needed context, and improves response speed and memory quality simultaneously.

**Maturity:** Shipped (Letta sleep-time agents; Honcho dreaming in production); underlying paper is peer-circulated research from April 2025

#### Zep/Graphiti: bi-temporal knowledge graph memory, now with embedded (serverless-friendly) backend and MCP server

Graphiti (getzep/graphiti, 28.6k GitHub stars, ~25k weekly PyPI downloads, v0.29.2 as of June 2026) ingests conversational 'episodes', LLM-extracts entities/relations, and stores them as a bi-temporal graph: every fact carries valid-time (when true in the world) and system-time (when learned); superseded facts are invalidated with validity windows, not deleted. Retrieval is hybrid: semantic + BM25 + graph traversal. #### Claude Code Agent Teams (peer-to-peer teammate sessions with shared task list + mailbox)

#### Judge-panel selection research: the 'selection bottleneck' result and multi-agent debate judges

2025-26 research pinned down when generate-then-select multi-agent pipelines actually work. 'When Agents Disagree: The Selection Bottleneck in Multi-Agent LLM Pipelines' (arXiv 2603.20324, March 2026) found across 210 tests / 42 tasks that diverse agent teams with judge-based selection hit a 0.810 win rate vs a single-model baseline, while homogeneous teams managed only 0.512 — and MoA-style synthesis/aggregation was preferred over baseline in zero of 42 tasks. Including weaker models in the pool paradoxically improved results while cutting cost; selector quality matters more than generator diversity. Complementary work: multi-agent debate judging with adaptive stability detection (NeurIPS 2025) formalizes when a judge panel's deliberation has converged, and Scorer/Critic/Commander adversarial-judge architectures reduce persuasion-style attacks on single judges; the Agent-as-a-Judge survey (arXiv 2508.02994) maps the space. Shipped analogue: Jules's Planning Critic (above) and Claude Code's adversarial-debate team recipe ('spawn 5 teammates to disprove each other's hypotheses') are these findings productized.

**Why it matters for this harness:** Hermes is self-improving — it generates its own skills, memories, and automations — so it needs a verification story for its own outputs. This research gives concrete design rules: (1) invest in the selector, not more generators — one good judge over N cheap diverse candidates beats fancy synthesis; (2) diversity only pays when selection is strong, which Hermes gets almost for free via multi-provider candidate generation (different models = decorrelated failures) plus one strong judge; (3) never merge/synthesize candidate outputs — pick one; (4) harden judges against persuasion with an explicit critic role. Applied to Hermes: skill self-improvement and trajectory curation for training data are both selection problems where a judge panel measurably beats a single pass.

**Maturity:** Research (arXiv March 2026; NeurIPS 2025) with shipped industrial analogues (Jules critic, Cursor multi-model best-of-n, Codex parallel attempts)

#### Anthropic's codified orchestrator-worker doctrine: when multi-agent pays (and when it doesn't)

### Tool-use advances for LLM agent harnesses (mid-2026): MCP spec evolution, programmatic/code-mode tool calling, tool search & deferred loading, structured outputs, parallel execution, computer-use & browser automation

#### MCP 2025-11-25 spec revision: async Tasks, URL-mode elicitation, OAuth Client ID Metadata Documents, sampling-with-tools

The Nov 25, 2025 MCP revision (shipped on MCP's one-year anniversary) added: (1) experimental Tasks (SEP-1686) — any request can return a task handle for call-now/fetch-later polling with states like working/input_required/completed; (2) URL-mode elicitation (SEP-1036) — servers can hand the client a URL for the user to complete a flow (e.g. OAuth consent) in a browser, plus richer enum elicitation schemas (SEP-1330) and defaults (SEP-1034); (3) OAuth Client ID Metadata Documents as the recommended client-registration mechanism (SEP-991) plus OIDC discovery and incremental scope consent via WWW-Authenticate (SEP-835); (4) tool calling inside sampling via tools/toolChoice params (SEP-1577); (5) JSON Schema 2020-12 as the default dialect, icons metadata, SSE polling (SEP-1699), and guidance that tool input-validation errors be returned as tool-execution errors so the model can self-correct (SEP-1303).

**Why it matters for this harness:** This is the current stable protocol Hermes should target as an MCP client. Tasks let Hermes kick off long-running server work and report back over chat asynchronously instead of holding a connection open (critical for serverless timeouts). URL elicitation is the missing piece for chat-platform UX: a server can ask Hermes to show the user a link (OAuth consent, payment) in Telegram/Discord, then resume. **Maturity:** Shipped, stable spec revision (modelcontextprotocol.io, open-governance project stewarded by Anthropic with formal SEP process and SDK tiering); Tasks marked experimental in this revision. Supported in official TS/Python SDKs.

#### MCP 2026-07-28 release candidate: stateless protocol, Multi-Round-Trip Requests, Tasks as official extension, sampling deprecated (bleeding edge)

The next MCP revision, RC locked May 21, 2026, final publishing July 28, 2026 (two weeks from now). Breaking changes: the initialize/initialized handshake and Mcp-Session-Id header are eliminated — client metadata travels in _meta on every request so any server instance can serve any request with no sticky routing (SEP-2567/SEP-2575). Server-initiated requests (elicitation, roots/list) are replaced by the Multi-Round-Trip Requests pattern: the server returns an InputRequiredResult with inputRequests + requestState, and the client retries with inputResponses (SEP-2322). Tasks graduate from experimental core to an official extension with tasks/get, tasks/update, tasks/cancel (tasks/list removed, SEP-2663). Sampling is formally Deprecated (SEP-2577) — replacement guidance is 'integrate with LLM provider APIs directly' — under a new lifecycle policy requiring 12 months before removal (SEP-2596). Extensions get reverse-DNS identifiers negotiated via capability maps. OAuth hardening: iss validation (SEP-2468), issuer binding (SEP-2352), application_type at registration (SEP-837).

**Why it matters for this harness:** Directly shaped for deployments like Hermes: stateless MCP means Hermes can run its MCP client (and any MCP servers it exposes) on serverless/Workers with zero session affinity, and MRTR gives a clean state-machine for pausing a tool call to ask the user something over chat and resuming later — a natural fit for asynchronous chat transports where responses arrive minutes later. The sampling deprecation validates Hermes's multi-provider design: servers should not expect to borrow the client's model. Harness authors should build against MRTR/Tasks now to avoid rework.

**Maturity:** Release candidate (final spec due July 28, 2026); shipped as RC by the MCP steering group. Old revisions remain supported during migration.

### Agent security and sandboxing: execution isolation (microVMs, gVisor, OS primitives), prompt-injection defenses, MCP/skill supply-chain security, secrets isolation, and egress control — state of the art as of July 2026

#### Anthropic sandbox-runtime (srt) — container-free OS-level sandboxing with proxy-enforced egress

Open-source (Apache-2.0) tool that sandboxes arbitrary processes without a container: bubblewrap + seccomp on Linux, sandbox-exec/Seatbelt profiles on macOS, and alpha Windows support via a dedicated srt-sandbox account with WFP egress filters. Filesystem is restricted to declared directories; the network namespace is removed entirely (Linux) and all traffic is forced through host-side HTTP/SOCKS5 proxies over Unix domain sockets, which enforce domain allowlists and prompt the user for new domains. Ships inside Claude Code's /sandbox feature (launched Oct 20, 2025); Anthropic reports it safely reduced permission prompts by 84%. Can wrap bash commands, whole agents, or individual MCP servers via a one-line change in .mcp.json. Actively developed: v0.0.65 released July 2026, 600+ commits. Anthropic's follow-up engineering post 'How we contain Claude' (May 30, 2026) documents the full containment stack: gVisor for claude.ai code execution, srt for Claude Code, full VMs (Apple Virtualization / Windows HCS) for Claude Cowork, and the principle that credentials never enter the sandbox (git creds live in an external proxy with scoped tokens).

**Why it matters for this harness:** This is the most directly reusable piece of tech for Hermes Agent: a VPS-deployable, no-Docker-required way to jail the agent's shell/tool execution and pin egress to an allowlist. Because Hermes is self-improving (it writes and runs its own code) and multi-provider, a harness-level OS sandbox that also wraps local MCP servers gives defense-in-depth that is independent of any model provider. The domain-allowlist proxy design doubles as the exfiltration chokepoint that prompt-injection defenses assume exists.

**Maturity:** Shipped in Claude Code (GA feature since Oct 2025); srt itself is a beta research preview, open-source, shipped by Anthropic

**Sources:** https://www.anthropic.com/engineering/claude-code-sandboxing · https://github.com/anthropic-experimental/sandbox-runtime · https://simonwillison.net/2026/May/30/how-we-contain-claude/

#### Hosted agent sandbox market: E2B (Firecracker) vs Modal (gVisor) vs Daytona (hardened containers) vs Cloudflare/Vercel Sandbox

Sandbox-as-a-service for agent code execution has commoditized with clearly differentiated isolation tiers. E2B: Firecracker microVMs, ~150ms boot, own kernel per sandbox, state held 1-24h by plan. Modal: gVisor user-space kernel, sub-second cold starts, and (uniquely among these) GPUs inside the sandbox at a ~3x price multiplier. **Why it matters for this harness:** Hermes Agent is multi-provider and runs on VPS/serverless; these services let it offload untrusted code execution (its own generated code, user-submitted code, skill testing) to a disposable, isolated environment via a simple SDK instead of hosting execution on the same box that holds its secrets and memory store. The isolation-tier taxonomy (container < gVisor < microVM) is the vocabulary to use when deciding what Hermes runs where.

**Maturity:** All shipped/GA commercial products; E2B ships Firecracker sandboxes, Modal ships gVisor sandboxes, Daytona ships hardened-container sandboxes, Cloudflare and Vercel ship their own sandbox SDKs

#### CaMeL and the 'deterministic policy outside the model' family of prompt-injection defenses

CaMeL (Google, Debenedetti et al., arXiv 2503.18813, v2 June 2025) defeats prompt injection by design rather than detection: a Privileged LLM sees only the trusted user query and emits a plan/program; a Quarantined LLM parses untrusted data but has no tool access; a custom interpreter tracks data provenance via capabilities and enforces security policies before every tool call. Result: 77% of AgentDojo tasks solved with provable security (vs 84% undefended). It anchors a now-recognized research family — FIDES, Progent, RTBAS, FORGE — that enforces security with deterministic reference monitors, capabilities, and information-flow labels outside the model. CaMeL is on the MIT 6.5660 security course reading list for 2026, and 2025-2026 evaluation papers (e.g. arXiv 2510.05244) show detector-style firewalls alone underperform these architectural defenses against adaptive attacks.

**Why it matters for this harness:** Hermes ingests untrusted content constantly (chat messages, web pages, emails) and is self-improving, so classifier-only defenses will eventually be bypassed. CaMeL's pattern is implementable at the harness layer Hermes controls: taint-track values that came from untrusted tool output, and require deterministic policy checks (or user confirmation) before those values can flow into consequential tool calls (send message, execute code, spend money). It is provider-agnostic — a pure harness architecture, not a model feature.

**Maturity:** Research with open-source reference code (google-research/camel-prompt-injection); the pattern is shipping in commercial reference-monitor products, but CaMeL itself is not a product

**Sources:** https://arxiv.org/abs/2503.18813 · https://css.csail.mit.edu/6.5660/2026/readings/camel.pdf · https://arxiv.org/pdf/2510.05244

#### LlamaFirewall (Meta) — open-source layered guardrail framework: PromptGuard 2 + AlignmentCheck + CodeShield

Meta's open-source security framework (arXiv 2505.03574, May 2025) combining three runtime guardrails: PromptGuard 2, a small BERT-based classifier screening inputs for jailbreak/injection patterns; AlignmentCheck, a chain-of-thought auditor that inspects the agent's reasoning trace for goal hijacking/misalignment (catches indirect injection after it lands); and CodeShield, an online static analysis engine that blocks insecure generated code. Combined layers reported ~90% attack-success-rate reduction (to 1.75%) in Meta's evaluations. Guardrails are customizable via regex or LLM prompts. Caveat established by 2026 benchmarks: firewall-style detectors are a useful layer but not sufficient alone against adaptive attacks.

**Why it matters for this harness:** For Hermes this is a free, model-agnostic screening layer to bolt onto the agentic loop: run PromptGuard-style classification on inbound chat-platform and web content, run AlignmentCheck-style trace auditing before high-consequence tool calls, and CodeShield-style scanning on self-generated code before Hermes executes or commits it — complementing (not replacing) sandboxing and CaMeL-style policies.

**Maturity:** Shipped, open source, by Meta; PromptGuard 2 models publicly released; actively cited as the reference 'agent firewall' in 2025-2026 literature

**Sources:** https://arxiv.org/abs/2505.03574 · https://arxiv.org/pdf/2510.05244

#### MCP supply-chain security: registry namespace authentication vs. real-world tool poisoning and registry poisoning

The attack side is now empirical, not theoretical: the first malicious MCP package in the wild (Sept 2025) masqueraded as a Postmark MCP server and BCC'd all email to an attacker; OX Security's April 2026 research poisoned 9 of 11 public MCP registries with a proof-of-concept package; MintMCP's 2026 analysis flags ~5.5% of public servers as tool-poisoning-vulnerable (adversarial instructions hidden in tool description fields that land directly in model context). On the defense side, the official MCP Registry (preview since Sept 8, 2025, still preview as of mid-2026) enforces namespace authentication — reverse-DNS names bound to verified GitHub accounts or DNS/HTTP domain-ownership challenges (e.g. com.example/server requires proving example.com) — but explicitly does not scan server code; emerging practice is signed manifests, version pinning, allowlists, and runtime monitoring of MCP servers, plus sandboxing each server (e.g. wrapping with srt).

**Why it matters for this harness:** Hermes has a skill/plugin/MCP surface and is self-improving, so it will install capabilities at runtime — making it exactly the target of tool poisoning and rug-pull attacks. Concrete harness implications: pin MCP server versions and hash-verify them, treat tool descriptions as untrusted input (diff them on update, alert on changes), only auto-trust namespace-verified registry entries, and run each third-party server inside its own sandbox with its own egress allowlist.

**Maturity:** Attacks: demonstrated in the wild (2025-2026). Registry namespace auth: shipped in preview by the MCP project (Anthropic-stewarded, community-run). Signing/scanning: ecosystem tooling still immature

#### Credential-broker egress proxies: Infisical Agent Vault (Apr 2026) and Pipelock agent firewall (2026) — secrets never enter agent context

Two 2026 open-source implementations of the same architectural consensus: the agent never holds secrets; a proxy outside its boundary injects them. Agent Vault (Infisical, research preview Apr 22, 2026, github.com/Infisical/agent-vault) is a TLS-terminating forward HTTPS proxy plus encrypted vault: the agent sets HTTPS_PROXY, makes normal requests, and the proxy strips any agent-attached credentials and injects the real one on a fresh upstream connection — structurally preventing credential theft via prompt injection; integrates with Claude Code, MCP tools, and plain HTTP clients. Pipelock (PipeLab, Apache-2.0, ~20MB Go binary, v2.3.0 by May 2026) is the complementary firewall direction: an 11-layer scanning pipeline on all agent egress with DLP for 48 credential patterns, 25 injection-detection patterns, SSRF protection, and rate limiting, speaking HTTP/CONNECT/WebSocket/MCP/A2A. Same pattern Anthropic describes for Claude Code on the web (git credentials held in an external scoped-credential proxy).

**Why it matters for this harness:** Hermes talks to many providers and services from a VPS, so its process environment is a honeypot of API keys — and any prompt injection that achieves code execution can read env vars. The broker pattern makes that a non-event: the sandbox holds no secrets and has no direct network; the proxy holds secrets and enforces per-host policy plus outbound DLP. This is arguably the single highest-leverage security upgrade for a self-hosted personal agent, and both tools are drop-in (HTTPS_PROXY) rather than requiring agent code changes.

**Maturity:** Agent Vault: open-source research preview shipped by Infisical (commercial secrets-management vendor). Pipelock: open-source Apache-2.0, v2.3.x shipped by PipeLab; both new in 2026

**Sources:** https://infisical.com/blog/agent-vault-the-open-source-credential-proxy-and-vault-for-agents · https://www.helpnetsecurity.com/2026/05/04/pipelock-open-source-ai-agent-firewall/ · https://pipelab.org/learn/ai-egress-proxy/

#### Agent Skills open standard (SKILL.md / agentskills.io) + skill marketplace ecosystem (skills.sh, SkillsMP)

#### SkillsBench — first benchmark measuring whether skills actually help (and showing self-generated skills mostly don't)

Released Feb 10, 2026 by the BenchFlow team (paper arXiv 2602.12670): 86 tasks across 11 domains, each a self-contained Docker container with deterministic programmatic verifiers (no LLM judging), run under three conditions — no skills, curated skills, self-generated skills — with full trajectory logging (7,308 trajectories, 7 agent-model configs). Headline results: curated skills improve pass rates by +16.2pp on average (from +4.5pp in software engineering to +51.9pp in healthcare), but self-generated skills yield negligible-to-negative gains — models either write imprecise procedures or fail to recognize when a skill is needed. Scoring follows the Terminal-Bench task-mean convention, and SkillsBench is integrated into rLLM's benchmark suite.

**Why it matters for this harness:** This is the single most load-bearing external result for Hermes: it directly stress-tests Hermes' headline feature ('creates skills from experience'). It provides (a) a ready-made harness for A/B-testing whether Hermes' skill-creation loop actually improves outcomes vs. the no-skill baseline, (b) evidence that naive one-shot self-generation underperforms — motivating Hermes' design of refining skills during use rather than only generating them, and (c) a regression suite to run when the skill-creation prompts change. Related 2026 follow-ons worth tracking: SkillGenBench (benchmarks skill-generation pipelines, arXiv 2605.18693) and SkillLearnBench (continual skill learning, arXiv 2604.20087).

**Maturity:** Shipped benchmark + paper (Feb 2026, BenchFlow / skillsbench.ai); already consumed by rLLM as a training/eval environment.

**Sources:** https://www.skillsbench.ai/blogs/introducing-skillsbench · https://arxiv.org/html/2602.12670v1 · https://www.skillsbench.ai/skillsbench.pdf

#### Trace2Skill and the 2026 trajectory-to-skill distillation lineage (batch distillation, not one-shot reflection)

Trace2Skill (arXiv 2603.25158, submitted Mar 26 2026, v5 Jun 4 2026, Jingwei Ni et al.) distills reusable SKILL-file-style skills by inductive reasoning over BATCHES of execution trajectories: it generates targeted patches from both success and failure cases and merges redundant fixes into a single conflict-free skill file. Distilled skills transfer across model sizes/families without retrieval or fine-tuning — skills evolved from Qwen3.5-35B trajectories improved a Qwen3.5-122B agent by up to +57.65pp on WikiTableQuestions. The surrounding 2026 lineage (descendants of Voyager's skill library): SkillRL (recursive skill-augmented RL — successful trajectories become strategy patterns, failures become 'lessons', code at github.com/aiming-lab/SkillRL), OPID (on-policy skill distillation from the policy's own rollouts, arXiv 2606.26790), SkillOS (learned skill curation, arXiv 2605.06614), CODESKILL (arXiv 2605.25430), and EvoSkills (co-evolutionary verification, arXiv 2604.01687).

**Why it matters for this harness:** Hermes currently creates skills after individual complex tasks (Voyager-style single-trajectory reflection). The 2026 consensus is that comparing MANY traces isolates reusable patterns from task-specific noise — single-run distillation captures noise. Hermes already has the exact prerequisites (batch_runner.py, trajectory_compressor.py, FTS5 session store), so a concrete upgrade path is: periodically distill/patch skills from batches of stored sessions, patching existing SKILL.md files (Trace2Skill-style merge) rather than accreting new ones. The cross-model transferability result also matters for a multi-provider agent: skills distilled on a cheap model's trajectories improve expensive models and vice versa.

**Maturity:** Research (multiple peer-track 2026 arXiv papers; SkillRL has released code; Trace2Skill on HuggingFace papers). Not yet productized, but directly implementable in a harness.

**Sources:** https://arxiv.org/abs/2603.25158 · https://huggingface.co/papers/2603.25158 · https://github.com/aiming-lab/SkillRL

#### Terminal-Bench 2.0/2.1 + Harbor — the de-facto standard harness-agnostic terminal-agent eval

Terminal-Bench 2.0 (Stanford x Laude Institute, launched Oct 2025): 89 hard, realistic end-to-end terminal tasks across 16 categories (SWE, security, scientific computing, data science, debugging...), each with a canary GUID against training contamination. Tasks are authored in the Harbor task format and executed by the Harbor harness, a framework for running agent evals at scale — any agent that can be installed in a container can be plugged in, which is why 51 agents across 17 scaffold types joined the leaderboard within 3.5 months. Current leaders: Codex CLI + GPT-5.2 at 63% average resolution, Terminus 2 + Claude Opus 4.5 at 58%. A 2.1 leaderboard is live at tbench.ai; Terminal-Bench 3.0 and a Terminal-Bench Science variant are in development; the TB paper was accepted at ICLR 2026.

**Why it matters for this harness:** Hermes is fundamentally a terminal agent (six terminal backends) and already ships mini_swe_runner.py — packaging Hermes as a Harbor-installable agent gives Nous (a) a public leaderboard presence comparable against Claude Code/Codex/Terminus, (b) a standardized regression suite for harness changes (new toolsets, prompt changes, skill-loop changes), and (c) the Harbor task format as a way to author Hermes-specific regression tasks (chat-platform flows, cron, subagents) that run in the same infra. SkillsBench deliberately reuses TB scoring conventions, so one integration covers both.

**Maturity:** Shipped and actively maintained (Stanford x Laude); 2.1 live, 3.0 in development; ICLR 2026 paper.

**Sources:** https://www.tbench.ai/ · https://arxiv.org/html/2601.11868v1 · https://snorkel.ai/blog/terminal-bench-2-0-raising-the-bar-for-ai-agent-evaluation/

#### rLLM v0.3 (Berkeley Sky Computing Lab) — RL training directly on existing agent harnesses, with Daytona/Modal sandbox support

rLLM is an open-source framework for 'agentic RL on any harness, with any backend, on any benchmark.' Its distinctive 2026 capability: it runs existing CLI agent harnesses AS-IS (Claude Code, Codex, mini-swe-agent, 10+ others, or custom agents) while automatically capturing token IDs and logprobs, structuring runs into Episodes/Trajectories/Steps, then applying RL (GRPO-family via verl, single-machine tinker, or Fireworks) — no harness rewrite required. Sandbox support: Docker, Daytona, Modal, local, with snapshot acceleration. 60+ integrated benchmarks including SWE-bench and SkillsBench. 2026 releases: rLLM UI (real-time observability for agent training, Mar 2026), on-policy distillation (train smaller students from stronger teachers, Mar 2026), v0.3.0-pre (Apr 30, 2026). Alternatives in the same space: OpenPipe ART (GRPO trainer where your app defines task+reward), SkyRL-Agent (arXiv 2511.16108, efficient multi-turn agent RL), AgentJet.

**Why it matters for this harness:** This is nearly a purpose-built match for Hermes' 'research-ready' pillar (batch trajectory generation + trajectory compression 'for training the next generation of tool-calling models'). rLLM's sandbox backends (Docker, Daytona, Modal) are literally the same execution backends Hermes ships, meaning Nous could train Hermes-series models by RL directly inside the Hermes harness — rewarding real tool-use trajectories rather than synthetic ones — and use on-policy distillation to compress a frontier-model-driven Hermes into a small open model. It also closes the loop the topic calls 'RL from agent experience': stored Hermes trajectories become training data with minimal glue.

**Maturity:** Open source, pre-1.0 (v0.3.0-pre, Apr 2026), maintained by Berkeley Sky Computing Lab; active monthly releases.

**Sources:** https://github.com/rllm-org/rllm · https://www.turingpost.com/p/agent-rl-training-tools · https://arxiv.org/html/2511.16108v1

#### GAIA2 + Meta ARE (and tau3-bench) — assistant-shaped benchmarks: dynamic, asynchronous, policy-constrained

Meta's ARE (Agents Research Environments, github.com/facebookresearch/meta-agents-research-environments) is an open platform for building dynamic agent environments; GAIA2 (dataset on HuggingFace; paper arXiv 2602.11964, original ARE paper arXiv 2509.17158) is its flagship benchmark: 800 verifiable scenarios (1,120 with augmentations) across 10 'universes' in a simulated Mobile environment with 101 tools. Unlike static benchmarks, the environment evolves independently of the agent — it tests temporal reasoning ('do X in 3 minutes'), asynchronous events, ambiguity handling, noise tolerance, and agent-to-agent collaboration. Complementary: Sierra's tau2-bench grew in 2026 into tau3-bench — five domains (airline, retail, telecom, banking-knowledge w/ configurable RAG), LLM-simulated users, full-duplex VOICE evaluation with real-time audio providers, policy-adherence scoring (booking the right flight but violating change-fee policy = fail), 75+ task quality fixes, leaderboard at taubench.com.

**Why it matters for this harness:** Almost all popular agent evals are coding-shaped; Hermes is a personal assistant on chat platforms with cron, reminders, memory, and multi-session continuity. GAIA2 is the closest public eval to that product: scheduled/async events map to Hermes' cron scheduler, the simulated mobile universe maps to its chat-platform tools, and time-based scoring punishes slow agentic loops. tau3-bench's policy-adherence + user-simulator methodology is the right template for regression-testing Hermes' conversational behavior (and its new voice evaluation matters given Hermes' voice-memo transcription). Both are runnable harnesses, not just leaderboards, so Hermes can be wired in as the system-under-test.

**Maturity:** Shipped: ARE/GAIA2 open-sourced by Meta FAIR (Sept 2025, 2026 paper update) with HF leaderboard; tau2/tau3-bench maintained by Sierra Research (MIT license), 38 model entries as of Apr 2026.

**Sources:** https://facebookresearch.github.io/meta-agents-research-environments/user_guide/gaia2_evaluation.html · https://huggingface.co/datasets/meta-agents-research-environments/gaia2 · https://arxiv.org/abs/2509.17158

#### HAL (Holistic Agent Leaderboard) + Docent — parallel eval infrastructure with rubric-based LLM-judge log inspection

Princeton's HAL (github.com/princeton-pli/hal-harness; paper arXiv 2510.11977) is an open-source, framework-agnostic harness that orchestrates agent evaluations in parallel across hundreds of VMs (weeks → hours), with cost tracking, three-dimensional analysis (model x scaffold x benchmark), validated via 21,730 rollouts across 9 models and 9 benchmarks (~$40k). Its most reusable piece for harness builders: automated log analysis with Docent — rubric-driven LLM-as-judge over full agent transcripts, scoring categories like instruction-following, tool use, verification behavior, self-correction, environmental barriers, and cheating/reward-gaming. Note the benchmark it wraps most, SWE-bench Verified, is now read as saturated (~5 models statistically tied near 80%), which is why Scale's SWE-bench Pro exists (Opus 4.5: 80.9% Verified vs 45.9% Pro on uncontaminated tasks) — pick discriminative benchmarks.

**Why it matters for this harness:** Two direct takeaways for Hermes: (1) HAL is a template (and usable tool) for cheap, parallelized, reproducible eval of Hermes across its provider matrix — a multi-provider agent needs exactly HAL's model x scaffold grid to know which providers regress; (2) the Docent pattern — rubric-based LLM-judge over transcripts — is implementable TODAY on Hermes' own FTS5 session store as a self-evaluation feature: nightly cron job judges yesterday's sessions for failed verifications, tool misuse, and unfinished tasks, feeding the memory/skill-improvement loop with graded rather than raw experience.

**Maturity:** Shipped open-source research infrastructure (Princeton PLI, Oct 2025 paper, actively maintained repo); SWE-bench Pro shipped by Scale AI with public leaderboard.

**Sources:** https://github.com/princeton-pli/hal-harness · https://arxiv.org/abs/2510.11977 · https://labs.scale.com/leaderboard/swe_bench_pro_public

#### CI regression gates with LLM-as-judge (Braintrust, LangSmith) + the separate-verifier-agent pattern

Production agent-eval platforms converged in 2025-26 on a specific regression-testing workflow: versioned eval experiments; a GitHub Action that runs the full eval suite on every PR and posts exact per-case score deltas as PR comments; regression gates that block merges below quality thresholds; and online scoring (LLM-as-judge + deterministic scorers applied to sampled production traces) to catch drift post-deploy. Braintrust and LangSmith both ship this; best practice per their 2026 guides is chain-of-thought judging (judge reasons through each criterion before scoring) which measurably improves human-correlation. In parallel, 2026 harness-engineering practice (Anthropic-adjacent 'loop engineering' writing, Addy Osmani's harness-engineering essay) has settled a related design rule for self-verification loops: don't let the acting agent grade its own work — spawn a separate read-only verifier agent holding the spec, since self-grading agents self-praise and the verifier, not the model, is the loop's bottleneck.

**Why it matters for this harness:** Hermes is uniquely exposed to silent regressions because it MODIFIES ITSELF (skills self-improve during use, memory accretes). A self-improving agent without regression gates can degrade unattended. Concrete adoption path: run a small eval suite (SkillsBench subset + Hermes-specific Harbor tasks) as a gate both in repo CI and — more novel — as a runtime gate whenever the agent edits one of its own skills: judge the patched skill against stored trajectories before accepting the edit, using a separate verifier subagent (Hermes already has subagent spawning). Braintrust/LangSmith are the managed options; the pattern is replicable with open tooling.

**Maturity:** Mature commercial products (Braintrust, LangSmith); the separate-verifier pattern is established practice documented across 2026 engineering writing, not a product.

**Sources:** https://www.braintrust.dev/articles/llm-evaluation-guide · https://www.braintrust.dev/articles/what-is-llm-as-a-judge · https://www.braintrust.dev/articles/agent-observability-complete-guide-2026

#### Letta sleep-time compute, memory models, and Context Repositories — productized memory distillation during idle time

Letta (the MemGPT team) shipped 'sleep-time compute': a two-agent design where a background sleep-time agent asynchronously reorganizes, consolidates, and rewrites the primary agent's memory blocks during idle periods, moving heavy memory reasoning out of the user-latency window (paper + code at github.com/letta-ai/sleep-time-compute). Their 2026 direction goes further: 'memory models' trained with memory generation as an explicit optimization objective (memories generated during sleep-time are scored by how much they improve the agent at test time — 'agent dreaming'), and the Context Repositories project (Feb 2026) rebuilds agent memory as programmatic context management with git-based versioning, with sleep-time workflows moving client-side so the agent reflects and consolidates using its own computer and tools. Letta is also an Agent Skills adopter (skills as one memory/capability substrate).

**Why it matters for this harness:** This is the shipped counterpart of the trajectory-distillation research, and it maps almost one-to-one onto Hermes' architecture: Hermes already does 'periodic nudges to persist knowledge' (lazy, in-conversation) and runs on serverless backends (Daytona/Modal) that hibernate when idle — a sleep-time consolidation pass triggered by Hermes' built-in cron before hibernation would upgrade memory curation from incremental nudges to deliberate distillation at near-zero marginal cost. Git-versioned memory (Context Repositories) also gives a concrete answer to the self-improvement safety problem: every memory/skill rewrite becomes a diffable, revertible commit — pairing naturally with the regression-gate pattern above.

**Maturity:** Shipped (Letta platform + open-source framework; sleep-time compute in production since 2025, paper + repo public); Context Repositories announced Feb 2026, in rollout; memory models research-stage at Letta.

**Sources:** https://www.letta.com/blog/sleep-time-compute/ · https://github.com/letta-ai/sleep-time-compute · https://www.letta.com/blog/towards-agents-that-learn/

### State-of-the-art LLM agent harness features 2025-2026 across Claude Code, Codex CLI, Gemini CLI, opencode, Amp, Cursor CLI, Aider

#### Claude Code Channels - chat platforms as push event sources into a running session

A 'channel' is an MCP server that pushes events (chat messages, webhooks, CI alerts) into an already-running Claude Code session via `claude --channels plugin:telegram@claude-plugins-official`. Official Telegram, Discord, and iMessage plugins ship in the research preview (requires v2.1.80+, plugins are Bun scripts). Two-way: inbound messages arrive as `<channel source=...>` events, the agent works against local files, then calls the channel's `reply` tool so the answer lands back in the chat app. Security: per-channel sender allowlists bootstrapped via pairing codes (`/telegram:access pair <code>`), org-level `channelsEnabled`/`allowedChannelPlugins` managed settings, and a 'permission relay' capability so tool-permission prompts can be approved from the chat platform. A localhost 'fakechat' demo and a channels-reference spec support building custom channels; non-interactive `-p` mode disables terminal-input tools so unattended sessions never stall.

**Why it matters for this harness:** This is Anthropic productizing Hermes Agent's core architecture (agent driven over chat platforms) as a harness primitive. Directly reusable design decisions: pairing-code + allowlist sender auth, permission-prompt relay over the same chat channel, reply-as-explicit-tool-call rather than routing raw model output, and packaging each platform bridge as a plugin against a documented protocol. Hermes could mirror the protocol shape or ship its own bridges as Claude-compatible channel plugins.

**Maturity:** Research preview, shipped 2026 by Anthropic in Claude Code (v2.1.80+); official plugin sources public in anthropics/claude-plugins-official

**Sources:** https://code.claude.com/docs/en/channels · https://github.com/anthropics/claude-plugins-official · https://code.claude.com/docs/en/changelog

#### Remote Control + Teleport - phone/web steering of a local agent with no inbound ports

Claude Code Remote Control (v2.1.51+, expanded to web+mobile in v2.1.198, July 2026) lets you drive a session running on your own machine from claude.ai/code or the Claude iOS/Android app. The local process makes outbound HTTPS only - it registers with the Anthropic API and polls; no inbound ports. **Why it matters for this harness:** Hermes runs on VPS/serverless and users control it remotely; this is the state of the art for that exact problem. The outbound-only relay-through-API architecture, QR pairing, worktree-per-session spawning, device-credential + biometric step-up trust model, and 'push when the agent needs a decision' notification taxonomy are all patterns Hermes can copy without any Anthropic dependency.

#### Plugin marketplaces - packaged, reviewed, versioned harness extensions (Claude Code plugins, Gemini CLI extensions)

Claude Code plugins bundle skills (namespaced slash commands), subagent definitions, hooks (hooks/hooks.json), MCP servers (.mcp.json), LSP servers (.lsp.json), background monitors (monitors/monitors.json - commands whose stdout lines are delivered to the agent as notifications), a bin/ dir added to Bash PATH, and default settings (which can even swap the main-thread agent). #### Checkpointing and /rewind - automatic undo of agent actions at conversation granularity

Claude Code automatically snapshots file state before each edit; every user prompt creates a checkpoint that persists across sessions (30-day retention, configurable). `/rewind` (or double-Esc) opens a menu listing each prompt with four actions: restore code+conversation, restore conversation only, restore code only, or 'summarize from here' to compress context. Limitation: only tracks edits made through the agent's file tools, not bash side effects (rm/mv/cp). The same capability is exposed programmatically in the Claude Agent SDK as file checkpointing. Codex CLI has an open feature request (openai/codex #12558) to clone the feature, and Gemini CLI has its own checkpointing; Cursor and desktop apps offer per-message checkpoints too.

**Why it matters for this harness:** A self-improving agent that edits its own code and config needs cheap, granular undo that is independent of git discipline. The design here - checkpoint keyed to user turns, separate restore of code vs conversation, and 'summarize from here' doubling as context management - is a proven UX Hermes can implement with a shadow git repo or content-addressed snapshots, plus the documented caveat that bash side effects need separate handling (e.g. sandboxed FS or fsnotify journaling).

**Maturity:** Shipped/GA in Claude Code and the Claude Agent SDK (Anthropic); Gemini CLI has checkpointing; requested-but-not-shipped in Codex CLI

**Sources:** https://code.claude.com/docs/en/checkpointing · https://platform.claude.com/docs/en/agent-sdk/file-checkpointing · https://github.com/openai/codex/issues/12558

#### Lifecycle hooks as a deterministic policy layer (Claude Code 12-event hooks; Codex hooks GA May 2026)

Claude Code hooks fire shell commands (or prompts to a lightweight model) at ~12 lifecycle events: SessionStart, UserPromptSubmit, PreToolUse, PostToolUse, Notification, Stop, SubagentStop, PreCompact, etc., with regex matchers on tool names. PreToolUse hooks receive tool_name/tool_input/tool_use_id as JSON on stdin and can return deny, ask (force confirmation), or allow-with-updatedInput (silently rewrite arguments). PostToolUse can inject feedback to the model. Hooks are distributable inside plugins. The Notification hook now carries agent_needs_input/agent_completed events for background subagents. OpenAI shipped hooks to GA in Codex CLI in May 2026, and Gemini CLI extensions can package hooks too - all three major harnesses have converged on this.

**Why it matters for this harness:** Hooks are how a harness enforces guarantees the model cannot: block edits to protected files, rewrite dangerous bash args, auto-format after writes, log every tool call for audit. For a self-improving agent this is the safety keel - the layer the agent itself cannot talk its way around. The deny/ask/rewrite tri-state and JSON-on-stdin contract are worth copying verbatim in Hermes.

**Maturity:** Shipped/GA: Claude Code (Anthropic, matured through 2025-2026); Codex CLI hooks GA May 2026 (OpenAI); Gemini CLI hooks via extensions (Google)

**Sources:** https://code.claude.com/docs/en/hooks · https://learn.chatgpt.com/docs/changelog · https://github.com/disler/claude-code-hooks-mastery

#### Background-by-default, nested subagents and massive orchestration (Ultracode workflows, Amp Oracle)

Claude Code v2.1.198 (July 2026) made subagents run in the background by default - the main agent keeps working and gets notified on completion. v2.1.172 (June 2026) allowed subagents to spawn their own subagents up to 5 levels deep. ### Inference and model routing for LLM agent harnesses (mid-2026): routing/cascades, reasoning-effort control, prompt caching economics, batch APIs, speculative decoding for agents, local serving, KV cache offload, telemetry standards, open-weight agentic models

#### Adaptive Thinking + effort parameter (reasoning-effort control replaces token budgets)

Anthropic's 4.6+ generation (Feb 2026) replaced manual `budget_tokens` with `thinking: {type: "adaptive"}` plus an `effort` parameter (low/medium/high, plus max on Opus 4.6; xhigh appears on Opus 4.7) placed in `output_config`. The model decides per-request whether and how much to think; at low effort it skips thinking entirely on simple queries. Adaptive mode auto-enables interleaved thinking between tool calls, and newer models (Fable 5, Mythos 5) ONLY support adaptive mode — sending budget_tokens returns a 400. Cache breakpoints are preserved across consecutive adaptive requests but broken by switching thinking modes; thinking blocks must be passed back unchanged in multi-turn tool loops. OpenAI's GPT-5.x line has the parallel `reasoning_effort` levels, so effort control is now a cross-provider abstraction.

**Why it matters for this harness:** Hermes Agent is multi-provider and pays per token: a single 'effort' knob per task class (chat reply on Telegram = low; skill-creation or coding subagent = high) is now the sanctioned way to trade cost/latency for quality, replacing brittle per-model token budgets. The harness must also handle the mechanics: pass thinking blocks back verbatim in tool loops, never flip thinking modes mid-conversation (kills prompt cache), and gate budget_tokens by model generation since new Claude models hard-reject it. This is exactly the kind of provider-divergent plumbing a providers/ abstraction layer needs to normalize.

**Maturity:** Shipped and GA: Anthropic API + AWS Bedrock (adaptive thinking, no beta header); budget_tokens formally deprecated on 4.6+. OpenAI reasoning_effort shipped across GPT-5.x.

**Sources:** https://platform.claude.com/docs/en/build-with-claude/extended-thinking · https://docs.aws.amazon.com/bedrock/latest/userguide/claude-messages-adaptive-thinking.html · https://claude5.ai/news/claude-sonnet-46-adaptive-thinking-replaces-extended-thinking

#### OpenRouter Auto Router with cache-aware session stickiness

OpenRouter's `openrouter/auto` meta-model (powered by NotDiamond) classifies each prompt and routes it to a curated pool of frontier models (Claude, GPT-5.x, Gemini, DeepSeek, gpt-oss-120b) with no routing surcharge. Key 2026 additions: implicit session stickiness — OpenRouter fingerprints the conversation and, once a provider reports prompt-cache usage, pins both model AND provider for that conversation so cache hits keep landing (5-min inactivity expiry, refreshed per request); explicit pinning via `session_id`; wildcard model restriction (e.g. `anthropic/*`); a 0-10 cost/quality dial. Underneath, provider routing sends requests to the cheapest reliable provider weighted by inverse-square of price, with user-controllable provider order, price ceilings, and fallback chains across 70+ providers. Academic backdrop: a 2026 survey (arXiv 2603.04445) formalizes the routing/cascading design space (pre-request rules, at-inference cascades, post-response retry).

**Why it matters for this harness:** Hermes already ships OpenRouter as a first-class provider, so this is nearly free leverage: routing cache-pinning solves the multi-provider agent's worst failure mode — a load-balanced router bouncing a long agentic conversation between providers and destroying prompt-cache economics (agent turns are 90%+ shared prefix). The session_id/stickiness mechanics are also a design template if Hermes builds its own routing layer: any model cascade in a harness must be cache-aware, i.e. route per-conversation, not per-request.

**Maturity:** Shipped: Auto Router GA with NotDiamond backend; session stickiness and curated-pool updates documented through June 2026. Routing/cascading survey is research (arXiv, 2026).

**Sources:** https://openrouter.ai/docs/guides/routing/routers/auto-router · https://openrouter.ai/blog/insights/model-routing/ · https://arxiv.org/html/2603.04445v2

#### Prompt caching economics have converged on 0.1x reads — and DeepSeek made it zero-config at 0.02x

As of mid-2026: Anthropic charges 1.25x for cache writes and 0.1x for reads (5-min TTL default, 1-hour option at higher write rate; any change to a block invalidates it and everything after); OpenAI's newer models cache automatically at 0.1x input with no write fee; Google implicit caching gives ~75% off; DeepSeek V4 does fully automatic disk-based context caching — no cache_control, no write fee, no storage fee — with cache hits at $0.0028/M on V4 Flash (98% off $0.14/M) and $0.0145/M on V4 Pro, over a 1M-token context window. Break-even math: Anthropic's 5-min tier pays for itself at ~1.4 reads per write; below ~3 hits/write OpenAI's no-write-fee model wins. Case study: ProjectDiscovery went from 7% to 84% cache-hit rate (59% total cost cut, 9.8B tokens served from cache) purely by moving dynamic working memory out of the system prompt to the end of the message list.

**Why it matters for this harness:** For a self-improving agent this is the single biggest cost lever. Hermes's learning loop mutates its own context (skills, memory nudges, user model) — if those mutations touch the system prompt or tool definitions, every turn is a cache miss at 10x the price. The discipline is: order blocks most-to-least stable (tools → system → reference/skills → history → live state), append-only conversation history, dynamic memory at the tail. On serverless (Modal/Daytona) where sessions hibernate, the 5-min TTL vs 1-hour cache tier choice is a real architectural decision. DeepSeek's zero-config disk caching plus 1M context makes it arguably the cheapest backend for long Hermes sessions.

**Maturity:** Shipped and GA across Anthropic, OpenAI, Google, DeepSeek; cross-provider cache pricing tracked live by Artificial Analysis.

**Sources:** https://artificialanalysis.ai/models/caching · https://ofox.ai/blog/prompt-caching-cost-math-anthropic-vs-openai-2026/ · https://api-docs.deepseek.com/quick_start/pricing/

#### Batch APIs at 50% off for the agent's offline half (Anthropic Message Batches, OpenAI Batch)

Anthropic's Message Batches API gives a flat 50% discount on all Claude pricing for async requests (up to 100,000 requests per batch, results within 24h, usually much faster) and stacks with prompt caching. OpenAI's Batch API is the same shape: 50% off, separate (much higher) rate-limit pool, 24h turnaround, up to 50,000 requests / 200MB input file. The economics only apply to latency-tolerant work — anything a user is actively waiting on is excluded.

**Why it matters for this harness:** Hermes has a clean split between interactive traffic (chat gateway) and offline work where batch pricing directly applies: batch trajectory generation for training data (batch_runner.py / research-ready pipelines), nightly cron automations (daily reports, weekly audits), session summarization for FTS5 cross-session recall, memory consolidation, and skill self-improvement passes. Routing that offline half through batch endpoints is a straight 50% cost cut with no quality tradeoff; combined with caching, the marginal cost of the learning loop drops dramatically. The harness needs a job-queue abstraction (submit, poll, deliver-to-platform) — which Hermes's cron scheduler already half-provides.

**Maturity:** Shipped and GA at both Anthropic and OpenAI for years, with 2026 limit increases; batch endpoints also offered by open-weight providers via OpenRouter.

#### Agent-specialized speculative decoding: Suffix Decoding (Arctic Inference) and EAGLE 3.1 in vLLM

Two complementary shipped techniques. (1) Snowflake's Arctic Inference vLLM plugin implements Suffix Decoding: draft-model-free speculation that matches the current generation against a suffix tree of previous outputs (~20µs/token on CPU). Because agentic output is highly repetitive (tool-call JSON, file paths, code edits), it delivers 1.8x-4.5x end-to-end SWE-Bench task speedup and 2.3x-6.3x faster decoding, enabled via vLLM's `--speculative-config` with `enable_suffix_decoding: true`. (2) EAGLE 3.1 (vLLM blog, May 26, 2026) fixes 'attention drift' at deeper speculation depths with FC normalization, giving up to 2x longer acceptance lengths in long-context workloads and 2.03x single-user output throughput — benchmarked on Kimi K2.6-NVFP4 with an open-sourced EAGLE 3.1 draft head, tool-calls included, shipping in vLLM v0.22.0.

**Why it matters for this harness:** When Hermes runs against self-hosted open weights (GPU cluster or a Nous Portal-style backend), decoding latency dominates the agentic loop — every tool call round-trips through generation. Suffix decoding is the notable one for harnesses because it needs no trained draft model and exploits exactly the repetition agents produce; it's a one-flag vLLM config change. The EAGLE 3.1 + Kimi K2.6 pairing shows the emerging pattern of frontier open-weight agentic models shipping with matched draft heads, which a harness's local-serving docs/presets should default to.

**Maturity:** Shipped: Arctic Inference open-sourced by Snowflake (2025) as a pip-installable vLLM plugin with pretrained speculators on Hugging Face; EAGLE 3.1 merged into vLLM main / v0.22.0 (May 2026). Speculative decoding itself ships in vLLM, SGLang, TensorRT-LLM, TGI.

**Sources:** https://vllm.ai/blog/2026-05-26-eagle-3-1 · https://www.snowflake.com/en/engineering-blog/fast-speculative-decoding-vllm-arctic/ · https://arxiv.org/pdf/2411.04975

#### LMCache: tiered KV-cache offload (GPU→CPU→NVMe→remote) validated on real agentic traces

LMCache is a KV-cache layer that extends vLLM/SGLang/NVIDIA Dynamo's in-GPU prefix cache with a storage hierarchy: CPU DRAM (L2), local NVMe (L3), Redis/remote object stores (L4), so KV for long conversations survives eviction and is reloaded instead of re-prefilled. A May 12, 2026 benchmark replayed 739 anonymized Claude Code conversation traces (20-150k input tokens/turn, 93-97% prefix reuse across turns) against MiniMax-M2.5 (230GB FP8 MoE) on 2x AMD MI300X with vLLM 0.19.0: under stress (32 users, 100k context) LMCache gave 3.0x lower average TTFT (34.6s vs 102.2s), 2.1x lower p95, 98% higher throughput (933 vs 471 tok/s), 2.3x more completed requests vs HBM-only.

**Why it matters for this harness:** The benchmark's core number — 93-97% prefix reuse across sequential agent turns — is the empirical case for why KV persistence matters to any harness: an agent's conversation IS a giant shared prefix. For Hermes self-hosting, LMCache means a hibernating serverless session or an idle multi-day Telegram conversation doesn't pay full re-prefill on wake; the KV comes back from CPU/NVMe. It's the self-hosted analogue of provider prompt caching, and it composes with Hermes's subagent model (parallel subagents sharing a system/tools prefix hit the same cache). Also proof that this works on AMD MI300X, widening cheap-GPU options.

**Maturity:** Shipped, open source, with an arXiv systems paper (Enterprise-Scale LLM Inference); integrated as the caching layer of the vLLM production stack; supports vLLM, SGLang, and NVIDIA Dynamo.

**Sources:** https://blog.lmcache.ai/en/benchmarking-lmcache-for-multi-turn-agentic-workloads-on-amd-mi300x/ · https://docs.lmcache.ai/getting_started/quickstart/offload_kv_cache.html · https://arxiv.org/html/2510.09665v2

#### OpenTelemetry GenAI semantic conventions v1.41: agent, tool, and MCP spans as the emerging cost/latency telemetry standard

The OTel GenAI semconv (now split into a dedicated repository, github.com/open-telemetry/semantic-conventions-genai) defines a vendor-neutral schema for LLM telemetry: client spans (`chat`, `generate_content`) with `gen_ai.provider.name`/`gen_ai.request.model`/token-usage attributes; agent-layer spans `create_agent`, `invoke_agent` (v1.41 splits CLIENT for remote vs INTERNAL for in-process frameworks), `invoke_workflow` (new in v1.41), and `execute_tool` (tool name now required in span name); MCP conventions since v1.39 (`tools/call` spans, `mcp.session.id`, W3C Trace Context across the protocol boundary); required metrics `gen_ai.client.operation.duration` and `gen_ai.client.token.usage`; v1.40-1.41 added cache-token and reasoning-token attributes and streaming metrics. Most gen_ai.* attributes still carry Development stability — names can change — with no published stabilization timeline as of May 2026.

**Why it matters for this harness:** Hermes spans exactly the layers this schema names — gateway → agent loop → subagents → tools/MCP → provider calls. Emitting v1.41-shaped spans gets per-turn cost attribution (including cached vs uncached and reasoning tokens — the two line items that dominate 2026 agent bills) into any OTel backend (Grafana, Datadog, Langfuse) with zero custom dashboards, and lets users compare providers apples-to-apples, which is the whole pitch of a multi-provider harness. Caveat for implementation: pin the semconv version and expect attribute renames until stabilization.

**Maturity:** Spec in active Development status (not yet stable) but versioned and widely implemented; v1.41 is current as of mid-2026. Shipped instrumentation: opentelemetry-instrumentation-openai-v2 (most mature), OpenLLMetry (Anthropic, Cohere, Bedrock).
