## Grounded proposals (top 13, verified against the source tree)

### A1. Code-Mode Tool Orchestration: full-registry programmatic tool calling with a persistent kernel

**Rank 1 — mean judge score 7.88 — category: tool-use power — effort: L**

**Summary.** Upgrade execute_code from 7 hardcoded stubs into a generated, typed Python API over the ENTIRE tool surface (builtin + plugin + MCP tools), so the model writes one script that searches, filters, joins, and loops over tools with intermediate results never entering context — plus an opt-in persistent interpreter session (notebook-style variable reuse across calls) and a structured artifact return channel (files/images, not just 50KB stdout). A user-visible leap: 'process these 2,000 rows across three MCP servers' becomes one turn instead of an unaffordable 100-iteration loop.

**Gap it fills.** Tool system subsystem: 'execute_code RPC surface is 7 hardcoded stubs (_TOOL_STUBS): MCP tools, plugin tools, and most builtins... are not scriptable; no persistent interpreter/session state across execute_code calls (fresh process each time); no artifact/image return channel other than stdout text.'

**Technology basis.** Anthropic Programmatic Tool Calling (beta advanced-tool-use-2025-11-20): 37% token reduction on complex research, ~38% billed-input reduction on 75-tool benchmarks (anthropic.com/engineering/advanced-tool-use); Cloudflare Code Mode: 2,500 endpoints exposed via search()+execute() in ~1,000 tokens, ~99.9% context reduction (blog.cloudflare.com/code-mode/); Microsoft Agent Framework CodeAct: 52.4% faster, 63.9% fewer tokens (devblogs.microsoft.com/agent-framework BUILD 2026). Provider-agnostic pattern — Hermes can run the loop locally against any model.

**What exists in the codebase today.** A solid PTC (programmatic tool calling) foundation exists, but the proposal's three pillars are all missing. What exists today: (1) tools/code_execution_tool.py (1,923 lines) implements execute_code with TWO transports — a UDS RPC server (_rpc_server_loop, line 491; loopback-TCP fallback on Windows) for the local backend, and file-based RPC (_rpc_poll_loop, line 767) shipped into remote terminal backends (Docker/SSH/Modal/Daytona/Singularity) via _get_or_create_env (line 631), which reuses terminal_tool's per-task_id persistent environment registry (_active_environments/_last_activity/_creation_locks/_resolve_container_task_id). (2) The scriptable surface is exactly the 7 hardcoded stubs in _TOOL_STUBS (line 227) gated by SANDBOX_ALLOWED_TOOLS (line 62): web_search, web_extract, read_file, write_file, search_files, patch, terminal. generate_hermes_tools_module (line 273) emits string-template stubs, not schema-derived ones. No MCP tool, no plugin tool, and no other builtin is callable from a script. (3) Each execute_code call spawns a FRESH child process with a fresh tmpdir (local, line 1212) or fresh sandbox_dir (remote, line 941) — no variable persistence across calls; only the surrounding terminal *container* persists. (4) Return channel is stdout-only: 50KB head+tail truncation (MAX_STDOUT_BYTES, line 75), ANSI-strip, secret redaction; no image/file/artifact return, despite the harness already supporting a `_multimodal` envelope (tools/registry.py _normalize_handler_result line 627; rendered in agent/tool_executor.py lines 996-1013) and a disk-spill persistence layer (tools/tool_result_storage.py). Also relevant existing infra the proposal can reuse: RPC dispatch already recurses through model_tools.handle_function_call (line 1026) so middleware/hooks/approvals fire on the real tool name; RPC threads are wrapped with tools/thread_context.propagate_context_to_thread so approval routing survives (#33057); a whole-script security guard runs first (tools/approval.check_execute_code_guard, called at code_execution_tool.py line 1160); the schema is rebuilt per-session listing only enabled sandbox tools (model_tools.py lines 456-465, build_execute_code_schema line 1814); and tools/tool_search.py already implements a deferred-tool catalog with search/describe/call bridging and session scoping (scoped_deferrable_names, resolve_underlying_call) — but only as MODEL-facing tools, not inside the sandbox. tools/registry.py ToolEntry (line 79) already carries read_only/destructive/idempotent flags, though they are sparsely annotated (6 read_only=True occurrences across 3 tool files; MCP tools get the conservative default destructive=True). The jupyter-live-kernel skill (skills/data-science/jupyter-live-kernel/SKILL.md) is prompt guidance for driving Jupyter via terminal, not an execute_code kernel. Docs confirm the 7-tool limit is the shipped contract (website/docs/user-guide/features/code-execution.md).

**Value proposition.** Hermes already sells execute_code as "collapse N tool calls into one turn," but the collapse only works for 7 builtins — the moment a workflow touches GitHub MCP, Home Assistant, kanban, browser, vision, or any plugin tool, the model falls back to one-context-round-trip-per-call, and tool_search's bridge still costs a full turn per invocation. This feature makes the ENTIRE registry (builtin + plugin + mcp-* toolsets) scriptable through the exact same dispatch path that already enforces approvals, guardrails, middleware, and redaction — so 'label 500 emails via MCP, cross-reference against a local CSV, open GitHub issues for the anomalies' becomes one execute_code turn with only the summary entering context. The persistent kernel removes the second tax: today a follow-up question re-fetches everything because script state dies with the process; notebook-style variable reuse keyed on the existing task_id environment registry makes iterative data work (the actual usage pattern behind the batch_runner/data-science skills) incremental. The artifact channel closes the loop with the harness's existing _multimodal envelope: a script can return a rendered chart or generated file as an image block the model can SEE, instead of a path buried in truncated stdout. Because Hermes runs the loop locally, this delivers Anthropic's PTC-style token savings (their published ~37% reduction) on every provider Hermes supports, not just Claude.

**Design sketch.** Four components, all deltas to existing files.

1. Registry-driven API generation (replaces _TOOL_STUBS). Extend generate_hermes_tools_module to build the sandbox module from registry snapshots instead of string templates: for each session-enabled tool, derive a typed Python signature from ToolEntry.schema["parameters"] (JSON-schema types → Python annotations, required→positional, defaults from schema; run through tools/schema_sanitizer first, as mcp_tool._convert_mcp_schema already does). Tool naming: builtins keep bare names; MCP tools use their already-registered prefixed names (mcp__server__tool via prefixed_mcp_tool_name, mcp_tool.py ~4480), exposed both flat and as `hermes_tools.mcp.<server>.<tool>` submodule attrs. Cloudflare-style scaling decision: do NOT inline hundreds of stubs — generate typed stubs only for the session's non-deferred tools, plus three generic RPC functions `search_tools(query)`, `describe_tool(name)`, `call_tool(name, args)` that reuse tools/tool_search.py's catalog builder and scoring (same scoping semantics as scoped_deferrable_names) so the sandbox can reach the long tail in ~constant schema tokens. The execute_code description (build_execute_code_schema) stays small: it stops enumerating tools and instead documents the module + search/call surface, keeping the 7-tool doc lines only as examples.

2. Full-surface dispatch with policy gates. Widen the allowed_tools computation in execute_code/_execute_remote from `SANDBOX_ALLOWED_TOOLS & session_tools` to `session_tools minus a recursion/interactivity denylist` (execute_code itself, delegate_task spawn recursion by default, clarify, model_tools._AGENT_LOOP_TOOLS, desktop-only tools). Dispatch stays exactly where it is — _rpc_server_loop/_rpc_poll_loop → handle_function_call — but must additionally forward session_id, enabled_tools, enabled_toolsets/disabled_toolsets (today it passes only task_id, line 592/874) so tool_search scoping, hermes_cli.middleware.apply_tool_request_middleware/run_tool_execution_middleware, and plugin pre/post hooks fire with full context. Per-tool policy: consult registry.get_operation_metadata (read_only/destructive/idempotent) to (a) auto-allow read_only tools, (b) route destructive tools through the existing approval context (already propagated into RPC threads via propagate_context_to_thread), (c) optionally expose a config knob code_execution.tools.{include,exclude} mirroring mcp_servers.*.tools filtering. Raise DEFAULT_MAX_TOOL_CALLS for scripted workloads (config already supports code_execution.max_tool_calls).

3. Opt-in persistent kernel. Add `session: str|null` and `reset: bool` params to the execute_code schema. When session is set, instead of Popen-per-call, spawn (or reuse) a long-lived child running a small REPL server: reads {code} messages over the existing RPC channel (or a second UDS/req-file pair), execs into a shared globals dict, streams stdout/stderr back, reports completion. Keyed on (task_id, session) in a module-level registry mirroring terminal_tool's _active_environments/_last_activity/_creation_locks pattern, with the same idle-cleanup thread and _resolve_container_task_id aliasing so delegate subagents inherit the right kernel. Remote backends: the kernel process lives inside the already-persistent env container (container_persistent=True default) under BaseEnvironment.get_temp_dir(); the parent's _rpc_poll_loop is started per call against the same kernel dir. Interrupt/timeout semantics reuse the existing _kill_process_group + tools/interrupt cooperative checks; kill only the in-flight exec, keep the kernel unless reset=True or timeout escalation.

4. Structured artifact channel. Inject HERMES_ARTIFACTS_DIR into the child env (scrubbed-env safe; local: subdir of tmpdir that survives collection, remote: sandbox_dir/artifacts pulled back before cleanup) plus a `save_artifact(path_or_bytes, name)` helper in the generated module. After execution, the handler scans the dir: images (png/jpg/webp, size-capped, downscaled like vision_tools) are returned as a `_multimodal` envelope — already normalized by registry._normalize_handler_result and rendered/persisted correctly by agent/tool_executor.py; other files are registered via the tool_result_storage pattern (preview + on-disk path the model can read_file). The registry entry's handler lambda (code_execution_tool.py line 1915) returns the dict instead of a plain JSON string; max_result_size_chars stays 100_000 for the text part.

Composition: no changes needed to the learning loop, gateway, or transports — everything rides handle_function_call, whose middleware/hook/approval chain is transport-agnostic; the gateway approval wait loop already handles execute_code (approval.py _await_gateway_decision shared with check_execute_code_guard). Toolset gating unchanged: toolsets.py code_execution toolset still owns the single execute_code tool. Tests extend tests/tools/test_code_execution.py, test_code_execution_modes.py, test_execute_code_approval_cluster.py.

**Effort.** L — the RPC transports, dispatch-through-middleware, env scrubbing, approval guard, and per-task environment lifecycle all exist; the work is schema→signature generation, allowlist/policy widening, a kernel process + registry, and the artifact scan/return path, each individually S/M but with heavy security-test surface.

**Risks.** Security is the dominant risk: widening from 7 curated tools to the full registry means LLM-generated code can loop destructive operations (MCP write tools, cronjob, discord) at machine speed; registry read_only/destructive flags are sparsely annotated (6 read_only=True across 3 files, MCP defaults to destructive=True), so policy gating starts conservative and depends on annotation debt being paid. Approval UX can degrade into prompt storms if each destructive call inside one script triggers a separate gateway approval — needs script-level batching or the existing whole-script guard to subsume per-call prompts. Schema fidelity: MCP JSON schemas (unions, anyOf, nested objects) don't map cleanly to Python signatures; a **kwargs fallback per tool is required or generation breaks on exotic servers. Persistent kernel adds lifecycle hazards — memory growth, leaked secrets accumulating in interpreter state across turns (redaction only covers stdout, not stored variables), stale kernels after env teardown, and interrupt semantics (kill exec vs kill kernel). Recursion loops (execute_code→delegate_task→execute_code) need explicit depth guards. Token-economics regression risk: naively inlining full-registry stubs would bloat the execute_code schema — the search()/call_tool() long-tail design must hold. Windows parity (TCP RPC transport) and remote file-RPC latency (one env.execute round-trip per tool call — already slow, worse at high call counts) may need a batched RPC frame.

**Judge panel notes.**
- Bulk multi-tool tasks collapse from 100-iteration slogs into one fast, cheap turn — users directly feel faster, cheaper sessions whenever they do real work across tools, though simple chats see less benefit.
- Generalizing _TOOL_STUBS (tools/code_execution_tool.py:227) to registry-derived typed stubs is a natural extension of an existing, working RPC mechanism; the registry already holds every schema. Persistent kernel and artifact channel add scope, and the file-based remote-RPC backends complicate persistence, but work stays confined to one module plus the registry.
- Code-mode over the full tool surface plus a persistent kernel and artifact channel matches the very latest frontier moves (Anthropic PTC beta, Cloudflare Code Mode) but done provider-agnostically and locally — level with the bleeding edge and ahead of every open harness.
- Programmatic access to the full tool registry is a cost-reducing substrate every future tool, MCP server, and workflow benefits from automatically (generated stubs from the registry keep maintenance sane), and the persistent kernel amplifies delegation, datagen, and automation alike.

### A2. Turn-Outcome Ledger: signal-driven reflection triggers and outcome-weighted skill utility scoring

**Rank 2 — mean judge score 7.50 — category: learning-loop instrumentation / skill curation — effort: L**

**Summary.** Persist the 8-state turn outcome (verified/blocked/failed/unresolved...) plus cost, retry counts, and guardrail halts as a per-turn row in the session DB, then close two loops with it. Reflection half: replace the blind every-N-turns memory/skill nudges with event-driven reflection — background review fires on failed/blocked/unresolved outcomes, detected user corrections ('no, I meant...'), repeated tool-failure streaks, and inbound emoji reactions from chat platforms (thumbs-down on a bot message = review this turn). Skill half: attribute each turn's outcome to the skills loaded that turn, accumulating per-skill utility scores (helped/neutral/hurt, cost-per-use); feed scores into the Curator's candidate table so consolidation/archival decisions weigh measured utility instead of just staleness, and rank/filter the system-prompt skill index by utility x relevance instead of listing everything alphabetically — cutting index tokens while surfacing skills that demonstrably work. Users get an honest answer to 'is my agent actually getting better' via queryable outcome trends.

**Gap it fills.** Learning-loop subsystem: 'Learning triggers are blind counters... no frustration/correction classifier, no thumbs-up/down capture'; 'No outcome/utility feedback loop: skill telemetry counts views/uses/patches but never whether a loaded skill actually helped... the curator judges by content overlap and staleness, never utility'; 'Insights are purely descriptive: nothing feeds cost-per-skill or load-but-never-used data back into curation'; fork delta: 'Turn outcomes are classified but not persisted as telemetry — the fork built both halves (outcome taxonomy, reviewer loop) without connecting them'; gateway: 'No inbound reaction events... a missed RLHF-signal channel, notable for a self-improving harness'. Verified: hermes_state.py has zero 'outcome' references today.

**Technology basis.** Google Jules' Planning Critic showing a 9.5% task-failure reduction from harness-level failure-signal loops (jules.google/docs/changelog); Braintrust/LangSmith online scoring of sampled production traces as the standard drift-catch pattern (braintrust.dev/articles/agent-observability-complete-guide-2026); Reflexion-style failure-triggered reflection now consensus in 2026 harness-engineering writing (addyosmani.com/blog/agent-harness-engineering); SkillsBench's programmatic-verifier methodology for measuring whether a skill helps (arXiv 2602.12670); SkillOS learned skill curation (arXiv 2605.06614) and OPID on-policy skill distillation's utility-scored skill selection (arXiv 2606.26790); Letta memory-models direction of scoring memories by measured downstream improvement (letta.com/blog/towards-agents-that-learn).

**What exists in the codebase today.** Three legs exist but are disconnected, exactly as the proposal claims — plus a few partial pieces the proposal under-credits. (1) Outcome taxonomy: /home/user/hermes-agent/agent/turn_outcome.py implements the full 8-state classifier (verified/completed_unverified/partial/blocked/failed/interrupted/unresolved/cancelled) with tests in tests/agent/test_turn_outcome.py. It is called at TWO finalizer sites, not one: agent/turn_finalizer.py:152 (chat-completions path) and agent/codex_runtime.py:503 (Codex app-server path — the proposal missed this second path). The outcome is returned in the result dict (turn_finalizer.py:436) and gates external-memory sync and background review, but is NEVER persisted: hermes_state.py has zero 'outcome' matches (verified via grep). (2) Reflection loop: agent/background_review.py has the full fork-and-review machinery (spawn_background_review_thread at line 943, three prompts: _MEMORY_REVIEW_PROMPT/_SKILL_REVIEW_PROMPT/_COMBINED_REVIEW_PROMPT), but triggers are blind counters — agent/agent_init.py:1359 sets _memory_nudge_interval=10 user turns (checked at agent/turn_context.py:306-314) and agent_init.py:1453 sets _skill_nudge_interval=10 tool iterations (checked at turn_finalizer.py:487-492, duplicated at codex_runtime.py:491-498). Critically, review fires ONLY when outcome=='verified' (turn_finalizer.py:506, codex_runtime.py:527) — failed/blocked/unresolved turns can never trigger learning today. No programmatic correction/frustration detector exists (the review PROMPT asks the reviewer to look for corrections, but the trigger is blind); agent/reactions.py has a tiny lexical detector but only for cosmetic affection signals ('ily', 'good bot'), wired UI-only at turn_context.py:322-334. (3) Skill telemetry: tools/skill_usage.py sidecar (~/.hermes/skills/.usage.json) tracks view/use/patch counts; _mutate() (line 579) loads the whole record dict, mutates in place, and saves — extra fields verified to round-trip, and get_record() backfills defaults via setdefault without dropping extras. skill_view bumps at tools/skills_tool.py:1742-1747. agent/curator.py renders these counters into the candidate list (_render_candidate_list, line 1472) but CURATOR_REVIEW_PROMPT rule 4 (line 450) explicitly disclaims them as weak evidence; apply_automatic_transitions (line 305) is pure staleness. agent/prompt_builder.py build_skills_system_prompt (line 1458) renders the index alphabetically (sorted at lines 1661/1673). agent/insights.py already mines per-skill skill_view/skill_manage loads from persisted messages.tool_calls (lines 300-365) — purely descriptive. (4) Inbound reactions: PARTIAL — Photon (plugins/platforms/photon/adapter.py:700-743) and Feishu (plugins/platforms/feishu/adapter.py:2606, registered at 1646-1651) already route reactions on bot messages into the agent as synthetic text events ('reaction:added:{emoji}'), gated to bot-sent targets. Slack receives reaction_added events but explicitly drops them (plugins/platforms/slack/adapter.py:1117-1127, comment: 'Hermes does not currently need to route them into the agent loop'). Telegram polls with Update.ALL_TYPES (adapter.py:2009) so reaction updates already arrive but no MessageReactionHandler is registered; Discord reactions are outbound-only (adapter.py:1942-1986). Outbound lifecycle reactions (eyes/check/cross) plus a coarse 3-state ProcessingOutcome hook exist in gateway/platforms/base.py:4018. (5) Persistence substrate: hermes_state.py SCHEMA_SQL (line 713) has sessions/messages tables with session-level cost/token aggregates and a messages.platform_message_id column (useful for reaction-to-turn correlation); schema changes are declarative via _reconcile_columns (line 1343) — adding a table/column is just editing SCHEMA_SQL. The proposal is the delta: persist the already-computed outcome, replace counter gates with signal gates, attribute outcomes to loaded skills, and feed utility into curator/index/insights.

**Value proposition.** This is mostly wiring, not building: the fork already computes an 8-state outcome every turn, already forks a reviewer agent, and already keeps a round-trip-safe per-skill telemetry sidecar — then throws the connective tissue away. Four sharpened payoffs specific to this harness: (1) The most informative turns are structurally excluded from learning — background review requires outcome=='verified' (turn_finalizer.py:506), so a failed, blocked, or unresolved turn can NEVER trigger reflection; the harness literally only studies its successes. (2) The curator's own prompt admits its evidence is weak — CURATOR_REVIEW_PROMPT rule 4 says 'DO NOT use usage counters... use=0 is not evidence a skill is valuable'; outcome-weighted utility replaces that stated deficiency with measured helped/hurt signal. (3) The system-prompt skill index is unconditionally alphabetical (prompt_builder.py:1661-1673), spending index tokens uniformly on skills that demonstrably never helped; the compact_categories names-only demotion machinery already exists and can be driven by utility instead of static config. (4) 'Is my agent getting better?' is unanswerable today despite hermes insights having the exact mining loop (per-skill loads from messages.tool_calls) and a CLI surface — one new table turns it into a queryable outcome-trend answer. Bonus: Slack currently receives and deliberately discards reaction events, and Telegram already receives reaction updates (ALL_TYPES) with no handler — the RLHF-signal channel is one registration away on both, with Photon/Feishu providing the in-repo precedent for routing them.

**Design sketch.** Six components, ordered by dependency. (A) Outcome ledger — add a turn_outcomes table to SCHEMA_SQL in hermes_state.py (session_id FK, turn_id, timestamp, outcome, outcome_reason, turn_exit_reason, api_calls, tool_iterations, retry_count, guardrail JSON, cost_usd_delta, token deltas, skills_loaded JSON, model, feedback nullable for reactions/corrections); the declarative _reconcile_columns path makes this a pure SCHEMA_SQL edit plus SessionDB.record_turn_outcome()/get_outcome_trends() accessors. Write from a new shared helper (agent/turn_ledger.py) called best-effort-guarded from BOTH finalizer paths — turn_finalizer.py (after classify at line 152, where api_call_count, agent._tool_guardrail_halt_decision, and session cost counters are all in scope) and codex_runtime.py (~line 503). Per-turn cost/token deltas: snapshot agent.session_estimated_cost_usd and token counters at turn start in turn_context.py, diff at finalize. ### A3. Durable Agent Execution: checkpointed turns, exactly-once tool effects, and a crash-surviving subagent fleet with worktree isolation and a chat-native dashboard

### A5. Event-Driven Automations with a Review Queue: reactions, webhooks, email, and PR events as first-class triggers

**Rank 5 — mean judge score 7.00 — category: automation reach — effort: L**

**Summary.** Extend the time-only cron stack into an event automation engine: inbound triggers from message reactions (thumbs-up-to-approve, emoji feedback), user message edits, generic webhooks, IMAP/email push, and GitHub PR/CI events fire jobs whose results land in a per-user review queue (accept / retry / discard from chat) instead of fire-and-forget delivery. ### A9. OS-sandboxed local execution with a credential-brokering, domain-allowlist egress proxy (bubblewrap + Landlock/seccomp, Seatbelt on macOS)

**Rank 9 — mean judge score 6.62 — category: sandboxing & secrets infrastructure — effort: XL for the full proposal (new backend + proxy + TLS-terminating broker + redaction + two-platform jail matrix + tests), but cleanly phaseable: value-based redaction alone is S, sandbox backend without broker is L, raw-tunnel egress proxy +M, TLS-terminating credential injection +L.**

**Summary.** Structurally jail the default local execution path without Docker, and make the jail's egress proxy double as a credential broker. Sandbox half: a new TERMINAL_ENV=local-sandboxed backend using bubblewrap + Landlock + seccomp on Linux (all available unprivileged on a bare VPS) and sandbox-exec/Seatbelt profile on macOS; filesystem writes confined to the workspace + declared dirs (making HERMES_WRITE_SAFE_ROOT enforcement structural instead of heuristic), network namespace removed with all egress forced through a host-side HTTP/SOCKS proxy over a Unix socket that enforces a per-session domain allowlist and prompts (via the existing gateway approval buttons) on new domains; same wrapper reusable to isolate stdio MCP server subprocesses. Anthropic reports this design safely cut permission prompts 84% in Claude Code — the approval regex library becomes defense-in-depth instead of the only boundary. Broker half: invert the current blocklist-scrubbing model — tool subprocesses (terminal, execute_code, MCP servers) receive zero real credentials; the broker proxy process outside the sandbox holds secrets (from the existing SecretSource registry) and injects them per-host on a fresh upstream connection when the child sets HTTPS_PROXY. Placeholder tokens handed to children are useless if exfiltrated. Adds value-based redaction as the complementary half: register the concrete secret values loaded from .env/auth.json/secret sources with agent/redact.py so any bare occurrence in transcripts, logs, or tool output is masked regardless of prefix shape. Also enables mid-session credential rotation and per-backend secret leases (SSH/Modal sync no longer ships ~/.hermes credentials wholesale).

**Gap it fills.** Execution environments & security subsystem: 'No OS-level sandboxing for the default local backend... the single biggest gap and the root of several below'; 'No built-in network egress policy for execution environments: docker_network is a boolean all-or-nothing'; 'terminal-backend isolation doesn't cover non-shell code paths (MCP subprocesses)'; SECURITY.md itself concedes every in-process guard is a heuristic. On secrets: env stripping is a hand-maintained blocklist ('_HERMES_PROVIDER_ENV_BLOCKLIST'), 'Redaction is shape-based only — no value-based masking of actually-loaded secrets', 'Secret lifecycle is startup-only... no short-lived credential minting or leasing per tool call', and 'SSH backend... pushes the host's ~/.hermes credentials/ onto every remote'. **Technology basis.** Anthropic sandbox-runtime (srt): bubblewrap+seccomp / Seatbelt, network namespace removal + Unix-socket proxy with domain allowlists, wraps agents and individual MCP servers, Apache-2.0, v0.0.65 July 2026 (github.com/anthropic-experimental/sandbox-runtime, anthropic.com/engineering/claude-code-sandboxing — which also documents the 'credentials never enter the sandbox' pattern for Claude Code on the web); Codex CLI's Landlock+seccomp/Seatbelt trio with deny-by-default network via codex-network-proxy (developers.openai.com/codex/agent-approvals-security); Infisical Agent Vault (open-source TLS-terminating credential proxy, agent sets HTTPS_PROXY, proxy strips agent-attached creds and injects real ones — 'structurally preventing credential theft via prompt injection', Apr 2026, github.com/Infisical/agent-vault); Pipelock agent firewall (11-layer egress DLP, 48 credential patterns, Apache-2.0, helpnetsecurity.com May 2026).

**Value proposition.** Hermes' own SECURITY.md declares the OS the only real boundary and every in-process guard a heuristic — then ships a default backend (TERMINAL_ENV=local) with no OS boundary at all, and points users at bring-your-own Docker/OpenShell. Value-based redaction (registering concrete loaded values with redact.py) closes the complementary gap redact.py currently has by design: an opaque token with no known prefix passes every regex today. **Design sketch.** PHASED, four components. A) Sandbox backend: new tools/environments/local_sandboxed.py with LocalSandboxedEnvironment(LocalEnvironment) — only _run_bash changes: wrap argv [bash, -c, script] in a platform jail prefix. Wire-up: new 'local-sandboxed' branch in _create_environment (terminal_tool.py:1423), _get_env_config (1256), check_terminal_requirements (~2814), config bridging in hermes_cli/config.py; the base.py snapshot/CWD/interrupt machinery is argv-agnostic and inherited unchanged; process-group kill keeps working (start_new_session preserved, bwrap in-group). CONNECT handler: resolve host → policy = website_policy blocklist (deny wins) + url_safety private-IP rules enforced at connect (kills rebinding) + terminal.sandbox.allowed_domains + session grants; unknown domain → tools/approval.py request_tool_approval(pattern_key='egress_domain:<host>'), riding the existing once/session/always allowlist and gateway approval buttons (submit_pending/resolve_gateway_approval) — no new UX. Phase 1 tunnels allowed hosts raw (no TLS termination). C) Credential broker (phase 3): per-host injection map built from agent/secret_sources registry + secret_scope.load_env_file + explicit config (host → header/env-var mapping); children that request passthrough vars get placeholder values; broker terminates TLS with a per-session CA (SSL_CERT_FILE set only in child env, never host-trusted) and rewrites placeholder credentials on the upstream connection; plugs behind credential_pool for rotation/leases; later, SSH/Modal file_sync gains a 'lease' mode that syncs placeholders instead of get_credential_file_mounts() originals. Composition: tool schemas unchanged, so learning loop/toolsets/transports see identical tools; denied egress returns an actionable error naming the approval it needs; domain grants persist via the same approval store all gateway platforms already render.

### A10. Sleep-Time Memory Consolidation ('Dreaming') Pass

**Rank 10 — mean judge score 6.50 — category: memory hygiene — effort: L**

**Summary.** A memory counterpart to the skill Curator: during detected idle windows, a restricted background agent reviews recent sessions plus MEMORY.md/USER.md, deduplicates and merges entries, detects contradictions (preferring newer provenance), expires stale facts to the archive, and distills unreviewed session episodes into durable entries — giving the builtin path the end-of-session extraction it currently lacks, at near-zero marginal cost since the box is idle anyway.

**Gap it fills.** Learning-loop subsystem: 'No memory hygiene beyond exact-string dedupe: no contradiction detection, no staleness/expiry review of memory entries (curator only handles skills), no periodic memory-consolidation pass mirroring the skill curator' and 'Builtin path has no end-of-session extraction... everything the model didn't proactively save is lost at session end'.

### A12. Verified Skill Edits: Regression Gate on Autonomous Skill Writes

**Rank 12 — mean judge score 6.38 — category: self-verification — effort: L**

**Summary.** Stop shipping unvalidated self-authored skills. **Value proposition.** Hermes' learning loop has a structural blind spot this fills exactly: the per-turn review fires only on outcome=="verified" turns that also hit the 10-iteration skill nudge, so the sessions richest in lessons — failures, interrupts, budget exhaustions, short gateway sessions — never contribute to the skill library, and no ledger even records which sessions were reviewed, so nothing can catch up later. Meanwhile every transcript is already durably stored and FTS5-indexed in state.db, i.e. the training corpus exists and is queryable but is never mined. The distiller adds the missing third tier to an already-proven two-tier loop: per-turn review (capture one session's lesson) → curator (reshape the library) → distiller (mine BATCHES of sessions per task type for success patterns and failure anti-patterns, contrastively, and patch them into the exact class-level umbrella SKILL.mds the curator already maintains). Because it emits patches to existing umbrellas rather than new one-off skills, it counteracts the noise/bloat failure mode the curator prompt itself calls a "FAILURE of the library", and it reuses the trusted safety plumbing wholesale: write-origin guards, read-before-write, fail-closed deletes, pre-run snapshots, dry-run reports, and the same idle/interval gate — so incremental risk is low relative to the learning coverage gained.

**Design sketch.** Five components, four new files plus small wiring edits. (1) OUTCOME LEDGER (prerequisite the proposal wrongly assumed exists): persist turn outcomes at the point they are already computed — agent/turn_finalizer.py:152 has _turn_outcome in hand; add a small hermes_state.py migration (via the existing _reconcile_columns/schema-version machinery) adding either sessions columns (verified_turns, failed_turns, last_outcome, last_outcome_reason) or a turn_outcomes table keyed (session_id, turn_id), plus reviewed/distilled watermark columns (distilled_at REAL, distill_run_id TEXT). Fallback heuristic for legacy rows: end_reason + message-level finish_reason. (2) BATCHER (new agent/skill_distiller.py): select candidate sessions from SessionStateDB.list_sessions_rich (exclude curator/fork platforms — forks never persist so contamination is already impossible; exclude parent sessions superseded by compression children via parent_session_id lineage) where distilled_at IS NULL and ended_at older than a settle window; group into task-type batches using (a) tool-usage fingerprint from messages.tool_name, (b) FTS5 search_messages queries derived from existing umbrella-skill descriptions (skills_list), (c) first-user-message keywords; cap batch size (e.g. 8-15 sessions) and batches per run (e.g. 3). (3) COMPACTION: per-session digest built by reusing background_review._digest_history logic (or a text-only convert_to_trajectory_format pass) — each session compressed to ~1-2k tokens with outcome label, first user message, tool sequence, error excerpts, final response head. (4) DISTILLER FORK: clone the curator's _run_llm_review pattern exactly — AIAgent(platform="distiller", quiet_mode=True, skip_context_files=True, skip_memory=True, max_iterations bounded), _memory_write_origin="background_review" so tools/skill_manager_tool.py guards fire (read-before-write via skill_view marks, fail-closed deletes), tool whitelist = skills toolset + optionally read-only session_search; model resolved from a new auxiliary.skill_distillation slot (register in hermes_cli/main.py _AUX_TASKS and web_server _AUX_TASK_SLOTS; auxiliary_client already generically supports auxiliary.<task>.*; copy curator's _resolve_review_runtime precedence). **Effort.** L — two new modules (distiller + state/ledger migration, roughly 800-1200 LOC plus prompt), aux-slot/config/CLI registration, and tests; not XL because the fork pattern, write guards, scheduling gates, snapshot/restore, and reporting are all proven curator code to clone, but not M because the outcome-ledger schema migration and the session grouper are genuinely new surface.

**Risks.** Two of the proposal's stated foundations do not exist and must be built or scoped out: the "outcome ledger" (turn outcomes are computed but never persisted — without shipping the ledger, failure/success contrast degrades to noisy end_reason heuristics and the SkillRL-style lessons-from-failures value collapses) and the "Verified Skill Edits regression gate" (no such component; the real gates are read-before-write and fail-closed deletes — adequate but weaker than the name implies, so a bad distillation patch lands unless dry-run/report review is used). Token cost: batches of transcripts are large even digested; runs on the user's own keys, so it must default OFF (matching curator.consolidate) with hard batch caps, limiting real-world adoption. Grouping brittleness: FTS5-keyword task-typing can produce incoherent batches and therefore confident-but-wrong umbrella patches; skill-description-seeded grouping mitigates but needs a min-cohesion threshold. Write contention: distiller, curator, and per-turn review all mutate ~/.hermes/skills — serialize distiller after curator in the same tick and take a curator_backup snapshot first. Privacy: transcripts contain user data/secrets and skills are shareable/publishable via the hub, so the prompt needs explicit redaction rules. DB scale: state.db can hold years of sessions; candidate selection must page, respect parent_session_id lineage (avoid double-counting compression children), and settle-window filter to skip live sessions.

**Judge panel notes.**
- Better skills distilled from batches of past sessions improves competence over weeks, but the user never sees the mechanism and the payoff is diffuse.
- The idle-triggered Curator is a direct architectural template (fork agent, tar.gz snapshots, structured classification), and FTS5 session search plus cron scheduling already exist; a batch distillation pass is essentially a new curator variant, cleanly additive.
- Batch trajectory-to-skill distillation is straight from Mar-Jun 2026 research (Trace2Skill, SkillRL) and shipped in no harness — implementing it would put the learning loop ahead of the field, though it depends on the outcome ledger existing.
- Batch distillation amplifies skill quality and provides the missing catch-up pass, but it's a downstream consumer of the outcome ledger and memory/eval infra rather than substrate itself; a periodic LLM job is manageable maintenance.
